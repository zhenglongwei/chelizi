# 维修方案与最终报价确认流程（双阶段报价 + 锁价）

版本：v2.0  
日期：2026-03-26  
基于：《启发》与全流程对齐修订；与 [docs/产品/支付与结算总览.md](产品/支付与结算总览.md) 口径一致

---

## 1. 业务背景

- **预报价**：竞价阶段，服务商仅依据车主上传的**照片与文字描述**在 `quotes` 中提交报价；用于列表比价与选厂，与到店后检测结论存在**合理差异**。
- **选厂成单**：用户选定维修厂后生成 `orders`，将当时报价**冻结为预报价快照**（`pre_quote_snapshot`，结构与 `quotes` 明细一致），`repair_plan` 可先等同于预报价或待最终报价覆盖。
- **最终报价**：车辆到店、检测完成后，服务商提交**最终报价**（含明细、金额、工期、质保；**每项须可带项目级质保月数**，与订单级质保策略一致）；**保险事故车**须绑定**定损单**材料（见第 8 节）。
- **车主确认与锁价**：车主在小程序内**点击确认**最终报价后，写入**最终报价快照**（`final_quote_snapshot`），进入**锁价**状态。**锁价后不得再向车主新增任何收费项目或费用**；施工过程中若必须增项，**额外成本由服务商自行承担**，平台可记录内部备注但不形成对车主账单。
- **与原 v1.0 差异**：取消「维修中不经材料、不经确认随意调价」；到店后采用**多轮报价**（`order_quote_proposals`）：**每一轮**均须上传**证明材料**，并须经**车主确认**（同意则锁价并更新快照；拒绝则回退后服务商可再发起新一轮）。**锁价后仍可向车主发起新一轮报价**（仍须新材料 + 再确认），全历史在**评价页与评价流公示**。锁价后**不得在未获新确认的情况下**向车主新增收费；若需争议处理，走客服/工单。

---

## 2. 流程概览

```
竞价 → 服务商提交预报价（quotes）
  → 车主选厂 → 订单生成，写入 pre_quote_snapshot
  → 到店检测
  → 服务商提交最终报价（可含定损单附件，保险必填）
       ↓
  final_quote_status = 待车主确认
       ↓
  车主小程序「确认最终报价」
       ↓
  写入 final_quote_snapshot，锁价；repair_plan 与之一致；可开工
       ↓
  维修中（仅允许不影响应收金额的工序记录，不向车主加价）
       ↓
  服务商提交维修完成（须选负责技师/负责人 + 凭证）
       ↓
  车主确认完工 → 评价（展示预报价 vs 最终报价）
```

## 3. 数据模型

### 3.1 订单表扩展（orders）

| 字段 | 类型 | 说明 |
|------|------|------|
| pre_quote_snapshot | JSON | 选厂时的**预报价**快照 `{ amount, items, duration, value_added_services?, remark? }`（质保仅 `items[].warranty_months`） |
| final_quote_snapshot | JSON | 车主确认后的**最终报价**快照，结构同上；**锁价后权威** |
| final_quote_status | TINYINT | 0=未发起最终报价 1=待车主确认 2=已确认（锁价） |
| final_quote_submitted_at | DATETIME | 服务商最近一次提交最终报价时间 |
| final_quote_confirmed_at | DATETIME | 车主确认时间 |
| loss_assessment_documents | JSON | **保险车**：定损单照片/PDF URL 列表及可选 OCR 结构化字段 |
| repair_plan | JSON | 锁价后与 `final_quote_snapshot` 对齐；锁价前可与预报价一致或占位 |
| repair_plan_status | TINYINT | **deprecated 语义收窄**：锁价流程下，仅用于「最终报价待确认」与历史单兼容；新单以 `final_quote_status` 为准 |
| repair_plan_adjusted_at | DATETIME | 可选；最终报价提交时间可写入或复用 `final_quote_submitted_at` |

**锁价后**：`repair_plan` = `final_quote_snapshot` 的维修方案部分；**禁止**调用会提高车主应收的 `repair_plan` 更新接口。

### 3.2 items 与项目级质保

与 `quote.items` 一致并扩展：`[{ damage_part, repair_type, parts_type?, price?, warranty_months? }]`  

- **warranty_months**：该项目质保（月）；**不设订单/快照顶层 `warranty`**，避免与分项歧义。
- **质保展示**：订单详情、评价、分享卡等仅以**逐条项目质保**为准。

### 3.3 维修完成凭证扩展（completion_evidence）

在既有 `repair_photos` / `settlement_photos` / `material_photos` 基础上增加：

| 字段 | 说明 |
|------|------|
| lead_technician | **必填**（锁价后完工时）：`{ id?, name, level?, source: 'technician_certs_index' \| 'manual' }`，与 `shops.technician_certs` 下拉选择或手工负责人一致 |

## 4. 接口设计（清单级，与实现迭代对齐）

### 4.1 服务商提交最终报价（含多轮）

- **非保险车**：每轮须提交 `evidence.photo_urls`（至少 1 张有效图 URL）。**保险车**：每轮须提交定损单等材料（规则同前），写入 `loss_assessment_documents` / `evidence`。

```
PUT /api/v1/merchant/orders/:id/final-quote
Body: { items, amount, duration, warranty, value_added_services?, evidence? (非保险必填 photo_urls), loss_assessment_documents? (保险必填), supplement_note? (保险选填，定损单列项不足时说明) }
```

- 仅 `status=1`（维修中）、已存在 `pre_quote_snapshot`、且**当前无待确认轮次**（`final_quote_status != 1`；若已建表 `order_quote_proposals` 还须无 `status=0` 行）可调用；**已锁价**（`=2`）仍可发起**下一轮**。
- 非保险须 `evidence.photo_urls`；保险须 `loss_assessment_documents.urls`；不足时可传 `supplement_note`。
- 成功后：`final_quote_status=1`，**写入** `order_quote_proposals` 新行，通知车主确认（文案含轮次序号）。

### 4.2 车主确认最终报价（锁价）

```
POST /api/v1/user/orders/:id/final-quote/confirm
Body: { approved: true | false }
```

- `approved=true`：`final_quote_status=2`，`final_quote_confirmed_at=now`，`final_quote_snapshot` 固化，`repair_plan` 同步；**更新 `quoted_amount` 为锁价金额**（或与 `final_quote_snapshot.amount` 一致），供结算与佣金口径使用。
- `approved=false`：当前待确认轮次标记拒绝；`repair_plan` / 展示回退至**上一轮已确认**快照或预报价；若无已确认轮次则回退预报价，`final_quote_status=0`；服务商可再提交新一轮（新材料）。

### 4.3 原「修改维修方案」接口

- **新单**：**禁用**可提高车主应收的 `PUT .../repair-plan`；或接口返回业务错误「请使用最终报价流程」。
- **历史单**：仅兼容已存在未锁价老数据，直至数据迁移完成。

## 5. 页面与交互

### 5.1 服务商订单详情（M07）

| 改动 | 说明 |
|------|------|
| 预报价展示 | 展示 `pre_quote_snapshot`（只读） |
| 最终报价 | 无待确认且维修中：可进入表单提交（含已锁价时的**再次提交报价**）；待确认：只读等车主；锁价后展示当前锁价快照 |
| 维修完成 | **须**选择负责技师/负责人 + 上传既有必填凭证；**仅** `final_quote_status=1`（有待确认的到店改价）时禁用完工；`=0`（以预报价为准）或 `=2`（已锁价）可完工 |

### 5.2 车主订单详情

| 改动 | 说明 |
|------|------|
| 两阶段说明 | 文案区分「选厂时的预报价」与「到店后的最终报价」 |
| 待确认 | `final_quote_status=1` 时展示最终明细 +「确认最终报价」按钮 |
| 锁价后 | 展示锁价提示与质保摘要（含**逐项目质保**） |

### 5.3 竞价报价页（车主）

- 保留并强化免责与**两阶段教育**：预报价可能变化；平台通过偏差统计、排序与合规约束乱价。

### 5.4 评价页

- **固定展示**（不必有差异才展示）：**预报价金额/摘要** 与 **最终报价金额/摘要**；若有明细差异，展示关键增删改摘要。详见 [docs/pages/05-评价页.md](pages/05-评价页.md)。

## 6. 评价中的方案对比

- **主展示**：预报价（`pre_quote_snapshot`）vs 最终报价（`final_quote_snapshot`）。
- **旧数据**：若仅有 `repair_plan` 与 `quote` 差异，可继续展示「下单时 vs 维修时」对比以兼容。

---

## 7. 通知

- 服务商提交最终报价 → `user_messages` + 订阅消息（若有）引导车主打开订单确认。
- 车主确认锁价 → 通知服务商可全面开工。

---

## 8. 保险事故车（定损单）

- **预报价**：与非保险相同。
- **最终报价**：**须依据定损单**；服务商上传定损单影像或 PDF；平台可 OCR 辅助结构化。
- **字段不足**：定损单无法覆盖平台模板字段时，**强制补充**说明或附件，一并写入 `loss_assessment_documents` / 快照备注。
- **风险提示**（上传页、确认页固定文案）：如虚增定损涉嫌违法、伪造材料后果自负等（法务定稿）。

---

## 9. 边界与异常

| 场景 | 处理 |
|------|------|
| 车主不同意最终报价 | 工单/客服；不通过「加价 repair_plan」绕开 |
| 锁价后服务商主张增项 | 不得向车主收费；内部记录 |
| 多轮报价 | 无轮次上限；**存在待确认轮次时**不可再提交新轮；已锁价（`final_quote_status=2`）仍可发起**下一轮**（新 `revision_no`、新材料、再确认）。全量记录在 `order_quote_proposals` |
| 配件类型 | 最终报价 items 内 `parts_type` 承诺后，施工中不得擅自降级（合规与评价核验沿用原规范） |

---

## 10. 实施顺序建议

1. 数据库迁移：`orders` 新增快照与状态字段；`completion_evidence` 约定 `lead_technician`  
2. 后端：提交/确认最终报价接口；选厂时写入 `pre_quote_snapshot`  
3. 服务商端：最终报价表单、完工技师选择  
4. 车主端：确认最终报价、文案与质保展示  
5. 评价页：双报价展示  
6. 关停或限缩旧 `repair-plan` 加价路径（兼容期后）

- **自有报价单拍照**：OCR/AI 结构化 → 映射标准字段；**缺失必填项则阻断并提示补充**。
