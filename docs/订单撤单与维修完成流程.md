# 订单撤单与维修完成流程

版本：v1.0  
日期：2026-02  
适用范围：车主端、服务商端、后台运维

---

## 1. 概述

本文档描述订单取消交易逻辑、维修完成凭证与材料审核的完整流程。

**基本原则**：仅车主可发起撤单，服务商不允许撤单。

---

## 2. 取消交易逻辑

### 2.1 规则

| 场景 | 条件 | 处理方式 |
|------|------|----------|
| A | 订单未完成（status!=3）且**无拆检留痕** | 车主可直接取消交易，立即生效 |
| B | 已存在拆检留痕（`order_offline_fee_proofs` 任意记录） | 禁止车主直接取消交易；需走“取消交易处置/结案”流程 |
| C | 订单已完成 (status=3) | 不可取消 |

**拆检留痕**：以 `order_offline_fee_proofs` 为准（车主/服务商上传拆检费收据、支付凭证等）。一旦存在留痕，视为已发生拆检或进入拆检费处置阶段。

### 2.2 数据变更

| 步骤 | 触发 | 数据变更 |
|------|------|----------|
| 直接取消交易（A） | 车主取消 | UPDATE orders SET status=4；UPDATE biddings SET status=0, selected_shop_id=NULL |
| 禁止直接取消（B） | 车主取消 | 返回错误提示；后台可在订单详情“取消交易处置”中结案留痕（写入 `order_lifecycle_events`） |

## 3. 维修完成凭证

### 3.1 触发时机

服务商提交完工材料时（发起 `1→2` 请求），可传 `completion_evidence`。**前置条件（库内已启用 `order_repair_milestones` 表时由接口强制）**：须已至少成功写入一条 **`after_process`（展示名「完工」）** 的过程留痕；仅凭维修前、维修过程、零配件验真等节点留痕**不会**触发完工提交流程。**竞价新单**（存在 `pre_quote_snapshot`）：`final_quote_status=0` 或 `=2` 均可提交完工；仅当 `=1` 时须等待车主确认后再提交完工材料。老单无预报价快照时仍兼容 `repair_plan` 流程。

### 3.2 凭证类型与要求（当前为引导，接口不强制）

| 类型 | 说明 | 要求 |
|------|------|------|
| 修复后照片 | 维修完成后的实拍图 | 选填 |
| 定损单/结算单 | 保险有定损单，否则为结算单 | **必传（至少 1 张）** |
| 物料照片 | 配件、材料等 | 选填 |
| 负责技师/负责人 | `lead_technician` | **不再在接口层强制**；历史数据可保留 |
| 配件验真方式 | `parts_verification` | **不再在接口层强制**；可在过程说明中文字描述 |

**材料审核（不阻塞提交，但会影响是否进入待验收）**：
- 服务商提交完工材料时会创建 `material_audit_tasks` 并异步审核（不占用服务商等待时间）。
- 订单**不会立即**进入 `status=2`（待验收）；会先保持 `status=1`（维修中）。
- 若缺失定损单/结算单、或 AI 无法从结算单识别最终金额、或结算单金额与系统金额不一致（阈值：差额 > 1 元且偏差 > 1%）：转后台人工审核并要求服务商补充/修正。
- 仅当 AI/人工审核通过后，系统才将订单推进到 `status=2` 并通知车主验收。

### 3.3 存储

存入 `orders.completion_evidence`（JSON）：

```json
{
  "repair_photos": ["url1", "url2"],
  "settlement_photos": ["url1"],
  "material_photos": ["url1"],
  "warranty_card_photos": ["url1"],
  "lead_technician": { "source": "technician_certs_index", "index": 0, "name": "张三", "level": "高级技师" },
  "parts_verification": {
    "methods": ["official", "qr_scan"],
    "note": "选 other 时的说明"
  }
}
```

`parts_verification.methods` 取值：`official` | `qr_scan` | `face_to_face` | `paper_proof` | `other`（选 `other` 时 `note` 必填）。也可仅传 `{ "not_provided": true }` 表示暂不向车主说明验真方式。

历史数据可选 **`warranty_card_photos`**：**纸质或其它质保凭证**照片（曾用于评价页补充展示）。**当前产品路径**：商户完工提交不再采集该字段；质保以订单方案 **`items[].warranty_months`** 与订单详情展示为准。

### 3.4 维修过程节点留痕（维修中，与完工凭证区分）

- **用途**：订单 `status=1`（维修中）时，服务商可按**固定节点枚举**多次上传过程照片与短说明，供车主在订单详情查看**时间线**；**不等同**于第 3 节「维修完成凭证」，也不替代定损结论。
- **存储**：独立表 `order_repair_milestones`（`milestone_code`、`photo_urls` JSON 数组、`note`、`created_at` 等）；同一订单同一节点可多次追加记录，按时间展示。小程序端支持**同一页为多个节点分别选图后一次性提交**（仍按节点逐条 POST），也可多次进入本页仅补传某一节点。
- **当前写入节点 code**（新单，服务端与小程序一致）：`before_process`（维修前）、`during_process`（维修过程）、`parts_verify_process`（零配件验真）、`after_process`（**展示名「完工」**，code 不变）。零配件验真使用 `photo_urls` + `parts_verify_note`；**不再**向 `during_process` 写入 `parts_photo_urls`（新单）。过程留痕为 `POST .../repair-milestones`；`completion_evidence` 仍为 `PUT .../status`（`1→2`）写入，二者存储分离。
- **历史节点 code**（库内旧数据仍展示）：`pre_clean_inspect`、`hidden_damage`、`parts_off`、`parts_on`、`mid_qc`、`pre_delivery_clean`（展示名见服务端 `getMilestoneLabel`）。
- **接口**：`POST /api/v1/merchant/orders/:id/repair-milestones`（鉴权商户）；车主侧随 `GET /api/v1/user/orders/:id` 返回 `repair_milestones[]`。
- **限制**：仅 `status=1`；**不因**材料审核任务 `pending` / `manual_review` 禁止上传过程节点（质检与里程碑留痕并行）。
- **通知**：每次写入后插入 `user_messages`（订单类），并尽力发送微信 **`user_order_update`** 订阅消息；站内信为兜底。跳转订单详情时可带 `anchor=milestones` 便于定位「维修进展」区块；车主须在订单详情等处**多次授权**「订单变更」模板以补充一次性订阅额度。

### 3.5 与评价的关联

- **定损单/结算单**：用于与**锁价后最终方案**对照；报价准确性以 **预报价↔最终报价** 为主（见《05-店铺综合评价体系》）
- **修复后照片**：对应评价模块 3 的「完工实拍图」
- **物料照片**：对应评价模块 2 的「维修过程」材料

车主评价时，上述凭证**自动预填**到对应评价项，**不可删除**，需明确标注来源（服务商上传/车主上传）；车主可**新增**照片，新增部分标注「车主上传」。

**照片对应关系**：

| 服务商凭证 | 评价对应 | 用途 |
|-----------|----------|------|
| 定损单/结算单 | 报价准确性 | 比较实际项目与预估报价的差别，计算报价准确性（保险才有定损单，否则用结算单） |
| 修复后照片 | 完工实拍图 | 模块 3 完工验收 |
| 物料照片 | 维修过程材料 | 模块 2 维修过程 |
