# 评价与点赞奖励 - 定期结算方案

> **定位**：补齐《03-全链路激励驱动体系》《04-点赞追加奖金体系》中**月结类**奖励（互动轨的常规点赞、转化轨的内容转化与事后验证等）与发放方式，明确结算周期与实现路径。**评价升级差额**（`upgrade_diff`）为**运营可选**能力，**默认关闭**（`settleUpgradeDiffEnabled`），**不对用户**宣导为「追评领钱」；与三轨并列的**「追评阶段现金 / 追评单独追加奖励」产品口径已取消**。
>
> **已确认**：每月 10 日结算，到账预留 1～3 个工作日，全部定期结算，用户须有详细清单；历史上「分阶段上线」指工程接入顺序，**不**表示「评价升级差额」默认对用户开放。

## 一、需纳入定期结算的奖励类型


| 奖励类型             | 来源文档                                 | 触发条件                                                           | 当前实现                |
| ---------------- | ------------------------------------ | -------------------------------------------------------------- | ------------------- |
| **常规点赞追加**（互动轨）  | 04 §3.1、`like_bonus`                 | 结算月内完工订单 **ΣC × interactionPoolShare**（默认 10%）为全站池，按各主评当月 **weight_coefficient 之和**占比分配（`settlement-service.settleLikeBonus`） | ✅ 已实现（月度结算）         |
| **内容转化追加**（转化轨）  | 04 §3.2、`conversion_bonus`           | 7 天窗口内点赞 → 用户成交 → 按决策权重分配                                      | ✅ 已实现（月度结算）         |
| **事后验证补发**（转化轨）  | 04 §3.3、`post_verify_bonus`          | 修车后回头点赞，满足 5 项判定                                               | ✅ 已实现（待结算表+月度发放）    |
| **评价升级差额补发**（可选） | 本文 §4.2 ①、`settleUpgradeDiffEnabled` | **默认关闭**；仅运营显式开启后由月度任务处理；**非**用户对「追评」的默认现金承诺                   | ✅ 已实现（代码路径+月度，默认不跑） |


**说明**：**基础轨**（首评主评）保持**实时发放**，不纳入定期结算。**追评**仅影响展示、风控与（若开启）差额逻辑，**不设**与基础/互动/转化并列的「追评单独追加奖励金」名目；现金增量统一归入**互动轨、转化轨**及可选 `**upgrade_diff`**。

## 二、结算周期方案对比

### 方案 A：每月固定日期定期结算（推荐）


| 项目       | 说明                                                                                      |
| -------- | --------------------------------------------------------------------------------------- |
| **结算日**  | 每月固定日期（如每月 5 日），结算上一自然月应得奖励                                                             |
| **核算范围** | 上月 1 日 00:00 ～ 上月最后一日 23:59 产生的：常规点赞追加、内容转化追加、事后验证补发；**以及**配置开启时的评价升级差额（`upgrade_diff`） |
| **发放时机** | 结算任务跑完后 1～3 个工作日内到账（可配置）                                                                |
| **优点**   | 规则清晰、便于对账、减少实时计算压力、利于风控事后复核                                                             |
| **缺点**   | 用户需等待至次月，体验略滞后                                                                          |


### 方案 B：实时结算


| 项目       | 说明                                                                                           |
| -------- | -------------------------------------------------------------------------------------------- |
| **结算日**  | 事件发生时立即计算并发放                                                                                 |
| **核算范围** | 点赞发生 → 常规点赞追加；成交发生 → 内容转化/事后验证；**若**运营开启 `upgrade_diff`，差额在规则认定后发放（可与月结或单独策略一致，**现行主路径为月结**） |
| **优点**   | 即时反馈，用户体验好                                                                                   |
| **缺点**   | 内容转化需等 7 天窗口期结束才能确定；事后验证需等完工后 30 天内点赞；实时计算与风控复核压力大                                           |


### 方案 C：混合模式


| 奖励类型     | 结算方式                                  | 理由                              |
| -------- | ------------------------------------- | ------------------------------- |
| 评价升级差额补发 | **默认不启用**；若启用，**可与月结一致**（降低对用户承诺「秒到」） | 产品默认关闭；历史上方案 C 的「实时差额」**非现行默认** |
| 常规点赞追加   | **月度**                                | 需汇总当月点赞，批量核算更清晰                 |
| 内容转化追加   | **月度**                                | 7 天窗口期 + 多评价权重分配，适合批量           |
| 事后验证补发   | **月度**                                | 需结合完工时间、浏览记录等，批量更易复核            |


### 方案 D：双周结算


| 项目      | 说明                                     |
| ------- | -------------------------------------- |
| **结算日** | 每月 15 日、月末，结算前两周应得奖励                   |
| **优点**  | 比月度更频繁，等待时间缩短                          |
| **缺点**  | 内容转化 7 天窗口、事后验证 30 天窗口与双周边界不天然对齐，规则略复杂 |

## 三、推荐结论与理由

**推荐采用方案 A：每月固定日期定期结算**，理由如下：

1. **与您判断一致**：定期结算更利于规则透明、对账清晰、风控复核。
2. **统一口径**：所有「滞后型」奖励（常规点赞、内容转化、事后验证；**及**运营开启时的评价升级差额）统一走月度结算主路径，用户心智简单：「每月 X 号发上月奖励」。
3. **技术实现**：单次定时任务批量跑完，便于日志、对账、异常重跑。
4. **合规与风控**：04 文档强调「滞后结算、违规追溯」，月度结算天然留出复核窗口。

**可选优化**：若运营**显式开启** `settleUpgradeDiffEnabled` 且希望差额更快到账，可再评估单独队列或 T+N；**默认**与点赞/转化同一月结任务，避免对用户形成「追评即秒到」的预期。

### 4.2 核算逻辑概要

**① 评价升级差额补发**（**默认关闭**，`settleUpgradeDiffEnabled`）

- **产品口径**：**无**「追评单独追加奖励」类目；追评参与整体重评后，**仅在本开关开启时**才可能产生 `upgrade_diff` 流水，且**不对用户**宣导为追评现金权益。
- **与三轨关系**：日常现金增量来自**互动轨**（如 `like_bonus`）与**转化轨**（`conversion_bonus` / `post_verify_bonus`）；`upgrade_diff` 为遗留/可选补差逻辑，**默认不参与**用户心智与对客文案。
- 查询（若开启）：上月内 `reviews` 有追评提交、且整体性重评已跑完、且业务规则认定「应补差额」的记录（实现可能仍参考库内字段，**不对用户展示等级**）。
- 计算：`补发 = 升级后规则总应发 − 已发`（以代码为准）。
- 约束：与附录 **A.4** 一致，纳入单笔订单**总硬帽**（默认税前合计 ≤ **80%·C**），与基础、点赞、转化等**同一套** `order-reward-cap-service` 汇总口径。

**② 常规点赞追加**

- 查询：上月 `review_likes` 中 `is_valid_for_bonus=1` 且 `created_at` 在上月内的记录；上月 `orders` 中 `status=3` 且 `completed_at` 落在上月的订单，逐单算 **C** 后求 **ΣC**。
- 公式：全站池 `pool = ΣC × interactionPoolShare`（默认 **10%**，`reward_rules.platformIncentiveV1.interactionPoolShare`）；每条主评税前份额 `bonusBeforeTax_i = pool × (weightSum_i / Σ weightSum)`，`weightSum` 为当月该评有效赞 **`weight_coefficient` 之和**（与 `settleLikeBonus` 实现一致）。**不设**单评单月上限；池内当月未分配部分**不结转**（无有效赞则整池不发）。
- **约束**：`like_bonus` **入账前不再**经 `clampPayoutToOrderHardCap` 按单笔 80% 压减；流水仍记 `like_bonus` 且计入 **A.4** 汇总（转化扣减、审计）。**基础轨**仍受 `complianceRedLine`（默认 **60%**·C）等与首评实时发放相关约束。转化轨仍为每单 **10%·C** 池（`conversionPoolShare`），与全站互动池**语义独立**。

**③ 内容转化追加**

- 查询：上月内完成的订单，回溯 7 天内点赞过相关评价的有效点赞，按决策权重分配内容转化奖金池（佣金 **10%**，与事后验证共帽，见 `reward_rules.platformIncentiveV1`）。
- 约束：单笔最多 10 条分配；分配结果仍计入 **A.4 总硬帽**（与基础、点赞等合计 capped），不是「超出 80% 另算」。

**④ 事后验证补发**

- 查询：上月内满足「下单前 7 天有浏览 + 同品牌同车系同类项目真实交易 + 完工后 30 天内点赞」的点赞。
- 补发：该笔成交未给其他评价分配过时，从佣金 **10%** 池全额补发。

## 附录 A：评价三轨奖金——参数、公式与算例（查阅用）

> **口径**：**硬帽/池**等常取「税后实发 + 同笔 `tax_deducted`」为税前还原口径，与 `order-reward-cap` 一致；**`transactions.amount` 为税后实发**。**订单实收佣金**记为 **C**，与 `reward-calculator.computeOrderCommissionAmount` 一致：C = M \times r，M 为订单计费有效金额，`r` 为 `commissionRepair` 解析出的佣金率。某类名目 `compliancePreTaxOnly` 为真时可 0 代扣，以规则为准。  
> **配置来源**：默认见 `web/api-server/constants/platform-reward-v1.js` 内 `platformIncentiveV1`；运营可在 `reward_rules` 中覆盖（合并规则见 `services/reward-rules-loader.js`）。  
> **代扣时点**：在**本笔产生 `transactions` / 余额增加**的时点按次预扣，**不是**在「提现成功」时。  
> **税务算法**：`web/api-server/utils/labor-remuneration-withhold.js`（居民劳务报酬预扣、减除与表二；旧式「>800 超出 20%」已弃用代码路径）。提现不重算同笔代扣；若以支付结算日为扣缴唯一基准须财务另确认。

### A.1 基础奖金（首评主评，实时发放）


| 符号/参数             | 含义             | 取值范围或来源                                                                                                                  |
| ----------------- | -------------- | ------------------------------------------------------------------------------------------------------------------------ |
| L                 | 维修复杂度 L1–L4    | 由报价/维修方案与 `reward_rules.complexityLevels` 关键词匹配；**不再**做配件数破格升档（`applyComplexityUpgrade` 为恒等）                         |
| `baseFixed`       | 复杂度对应基础固定额     | `reward_rules.baseReward[L]` / `baseRewardInsurance[L]`，缺省见代码 `BASE_REWARD`                                              |
| `vehicleCoeff`    | 车价校准乘子         | 后台车价分档表 `getVehicleCoeff`；**平台化长期目标**为「车价只做分档输入、一档一额」，当前实现仍为**乘子**                                                       |

| `capByComplexity` | （已废止）按复杂度封顶    | **当前实现不参与**；库内字段可保留为 0，见 `normalize-reward-rules-db.js`                                                      |

| `maxByCommission` | 佣金红线封顶         | C \times \dfrac{\texttt{complianceRedLine}}{100}，`complianceRedLine` 默认 **60**（`platform-reward-v1.js` 根级与 `reward_rules` 可覆盖；`baseInteractionCapPct` 为文档/规划子帽叙事，默认与之一致） |
| B                 | 实发基础（首评入账基数之一） | B = \min(B_{\text{raw}},\ \texttt{maxByCommission})，元保留两位小数（与 `reward-calculator.calculateReward` 一致）                    |


**首评现金（无内容优质加项）**：**已取消**据补充说明 / AI 内容档位叠加的 **`premiumFloat`**；首评 `rebate` 以 **B**（及既有封顶、账号等级、硬帽）为上限，**不再**因「优质」+50%（`calcPremiumFloatReward` 仍保留在 `reward-calculator.js` 供历史兼容，评价提交流程**不调用**）。

**与三轨关系**：基础轨对应上表 **B**、经账号等级（0 级不发）与首评发放逻辑后的**实时 rebate**。

### A.2 互动奖金（以「常规点赞追加」为主的月结实现 + 规划项）

**阅读顺序**：若符号表不易理解，可先读 **A.2.1a 算例**（按真实结算顺序从点赞到入账），再回来看 A.2.1 表格与 A.2.2 规划项。

#### A.2.1 当前月结：`like_bonus`（`settlement-service.settleLikeBonus`）


| 符号/参数                 | 含义               | 取值范围或来源                                                                                                                                                                                |
| --------------------- | ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ΣC`                  | 结算月内完工订单的实收佣金之和 | `orders.status=3` 且 `completed_at` 落在结算月；每单 **C** 与 A.1 同口径（`computeOrderCommissionAmount`）                                                                                          |
| `pool`                | 互动全站池（税前）        | `pool = ΣC × interactionPoolShare`，默认 `interactionPoolShare = 0.1`（`platformIncentiveV1`）                                                                                           |
| `weightSum`           | 该主评在结算月内有效点赞的权重和 | \sum \texttt{weight\_coefficient}，仅 `is_valid_for_bonus=1` 且 `like_type='normal'`                                                                                                        |
| `ΣW`                  | 当月所有参与分配主评的权重和之和 | 各主评 `weightSum` 之和（仅 `weightSum>0` 的主评）                                                                                                                                           |
| `bonusBeforeTax`       | 该主评当月互动税前应发       | `pool × (weightSum / ΣW)`，元保留两位；末条用「池 − 已分摊」消除舍入差                                                                                                                                   |
| 个税预扣                 | 居民劳务报酬按次（减除+表二） | 与 `labor-remuneration-withhold.js` 一致；`compliancePreTaxOnly` 为真时名目共 0 代扣的仍以规则为准；`like_bonus` 以结算任务实算为准 |
| **按单硬帽**              | 互动入账           | **不**再调用 `clampPayoutToOrderHardCap`；`like_bonus` 仍计入 A.4 类型汇总，供转化侧扣减已发互动等审计 |

#### A.2.1a 算例：`like_bonus` 从数据到入账（虚构，与 `settleLikeBonus` 一致）

**场景**：结算月内全站仅两单完工，**C₁=600、C₂=400**，故 **ΣC=1000**。`interactionPoolShare=0.1` ⇒ **pool=100** 元。全站该月仅两条主评有有效赞：`REV_A` 的 **weightSum_A=3.5**，`REV_B` 的 **weightSum_B=1.5**，**ΣW=5**。

**第 1 步：哪些点赞会计入**

- 只取 `review_likes` 中：`like_type = 'normal'`、`is_valid_for_bonus = 1`、且 `created_at` 在**结算月**内的行。

**第 2 步：权重和**

- 与旧方案相同：每条赞落库时的 **`weight_coefficient`**（账号可信度 × 车型匹配等，见 **A.2.1b**）。  
- **REV_A**：**weightSum_A = 3.5**；**REV_B**：**weightSum_B = 1.5**。

**第 3 步：税前分摊**

- **REV_A**：`100 × 3.5 / 5 = 70.00` 元。  
- **REV_B**：`100 − 70 = 30.00` 元（末条吃舍入余量，与代码一致）。

**第 4 步：劳务报酬预扣（与首评、转化等同套工具函数）**

- 以 `bonusBeforeTax` 为单次收入，按「**≤4000 减 800；>4000 减 20%**」得预扣应纳税所得额，再按**表二**计 `tax_deducted`；`bonusAfterTax` 入账。小额可能税额为 0，并非仅以「是否超过 800」为界。

**第 5 步：入账**

- 写入 `transactions.type = 'like_bonus'`，`settlement_month` 为结算月，`reward_source_order_id` 锚定该评所属订单（与现表结构一致）。

**小结**：月结互动轨为 **全站池按权重占比分配**；**不再**使用「单订单 C×0.5%×权重和」公式，也**不再**在互动路径上按单笔 **80%·C** 压减。

#### A.2.1b 有效赞（`is_valid_for_bonus`）与权重系数（实现口径）

与《04》一致，**是否在月结里算钱**在**点赞落库时**就写进 `review_likes.is_valid_for_bonus`（0/1），月结任务只读该字段，**不再重算**。

**1）怎样算「有效赞」（可纳入 `like_bonus` 权重和）**

同时满足（见 `web/api-server/services/review-like-service.js` → `likeReview`）：

| 条件 | 说明 |
| ---- | ---- |
| 累计有效阅读 ≥ **10 秒** | 来自 `review_reading_sessions` 对该 `review_id + user_id` 的 `effective_seconds` 汇总；单条会话最多计 **180** 秒/次，单人单评累计上限 **300** 秒（防刷） |
| 点赞者**可信度权重 > 0** | 由 `antifraud.getUserTrustLevel` 得到等级 **1～4**；**0 级**可信度权重为 **0**，则 `is_valid_for_bonus = 0`（点赞仍可成功，但不进奖金） |
| 类型为常规月结 | 月结 SQL 使用 `like_type = 'normal'`；若同一赞同时满足事后验证规则，会标为 `post_verify`，**不走**常规 `like_bonus` 汇总（走事后验证链路） |

不满足时：`is_valid_for_bonus = 0`，前端可提示「有效阅读不足 10 秒，不纳入奖金核算」等（与接口返回一致）。

**2）权重系数 `weight_coefficient` 怎么算**

仍由 `likeReview` **在落库时**计算并写入：

- **基础可信度权重**（点赞者账号等级，与《01》6.2 一致）：0→**0**，1→**0.3**，2→**1.0**，3→**1.5**，4→**2.0**。  
- **车型高权匹配**：比较点赞者画像车与**被赞评价所在订单**车（`bidding.vehicle_info` 的 **brand + model**）：双方经规范化后得到同一 **`品牌|车型` 键** 且**键非空**则记 **1**，否则 **0**。点赞者画像优先 **`user_vehicles`**，若缺品牌或车型则回退该用户**最近完工订单**的车辆信息；**不比较车牌**，避免夫妻同车双账号因同牌获得不当高权。库表列名仍为 `vehicle_match_by_plate`，语义为「同车型高权」，见迁移 `migration-20260414-review-likes-model-match-comment.sql`。  
- **综合系数** = `基础可信度权重 × 车型匹配乘子`，其中乘子：**同品牌同车型（高权）= 2.0**，**否则 = 0.5**；结果保留四位小数：`round(乘积, 4)`。

**算式**：`weight_coefficient = round( credibilityWeight × (vehicleMatchByModel ? 2.0 : 0.5), 4 )`。

**例**：点赞者为 **3 级**（可信度 **1.5**），且与评价订单**同品牌同车型**，则 **1.5 × 2.0 = 3.0**；若为 3 级但**未形成同一车型键**（不同车型、或任一侧缺品牌/车型），则 **1.5 × 0.5 = 0.75**。若为 **0 级**，可信度为 **0**，则权重为 **0** 且本条**不会** `is_valid_for_bonus=1`。

#### A.2.2 规划常量（互动质量分 Q_m，`platformIncentiveV1.interaction`）

> **说明**：下列含规划项 **Q_m**；**现行 `like_bonus` 现金**以 **A.2.1 全站池 + 权重占比**为准。`baseInteractionCapPct` 与根级 `complianceRedLine` 对齐，用于基础轨封顶叙事。


| 参数                                                | 含义                                | 默认       | 建议可调范围（计划 §十）  |
| ------------------------------------------------- | --------------------------------- | -------- | -------------- |
| `interactionPoolShare`                            | 互动全站池占当月 ΣC 比例                  | **0.1**  | 与财务对表可调        |
| `baseInteractionCapPct`                           | 文档/规划：基础子帽占 C 比例 P/C（%）         | **60**   | 与 `complianceRedLine` 一致 |
| `s_E`                                             | 阅读率贝叶斯平滑「曝光先验」                    | **5**    | 3～12（次）        |
| `s_R`                                             | 阅读率贝叶斯平滑「阅读秒先验」                   | **300**  | 180～480（秒）     |
| `rhoCap`                                          | 平滑读率 \hat\rho 上限                  | **0.5**  | 0.35～0.55      |
| `tau`                                             | 单月发放不超过 \tau \cdot I_{\text{cap}} | **0.25** | 0.15～0.35      |
| `eta.E` / `eta.R` / `eta.L` / `eta.C` / `eta.rho` | Q_m 中曝光/阅读/赞/评/读率项权重              | 见常量文件    | 同量级标定，避免单维过大   |
| `coldStartD`                                      | 支付式 Q/(D+Q) 的冷启动尺度 **D**          | **50**   | 与 \lambda 联动标定 |


**规划公式摘要**（单主评、单月，自单）：

- P = C \times \dfrac{\texttt{baseInteractionCapPct}}{100}，I_{\text{cap}} = \max(0,\ P - B)（B 为已发基础实发口径）。
- \hat\rho = \min\left(\texttt{rhoCap},\ \dfrac{R_m + s_R}{E_m + s_E}\right)（防 E_m=0 用先验）。
- Q_m = \eta_E\ln(1+E_m) + \eta_R\min(R_{\text{cap}}, R_m) + \eta_L \tilde L_m + \eta_C\min(C_{\max},C_m) + \eta_\rho \hat\rho（R_{\text{cap}},C_{\max} 由产品取上限）。
- 月发 `pay_m`：`min(Rem,\ \min(I_{\text{cap}},\ \lambda Q_m))` 或 `min(Rem,\ I_{\text{cap}}\cdot \min(1,\ Q_m/(D+Q_m)))`，且 `pay_m ≤ tau * I_cap`。

### A.3 转化奖金（内容转化月结 + 与事后验证共池）


| 符号/参数                  | 含义                          | 取值范围或来源                                                                                              |
| ---------------------- | --------------------------- | ---------------------------------------------------------------------------------------------------- |
| \alpha                 | 转化池 + 事后验证**共用**帽占 C 比例     | `**conversionPoolShare`**，默认 **0.1**（10%）                                                            |
| P_O                    | 归因订单上的池（税前）                 | P_O = C \times \alpha                                                                                |
| L_B                    | 买家可信度等级 0–4                 | `antifraud.getUserTrustLevel`                                                                        |
| \varphi(L_B)           | 买家等级对整池缩放 `**phi[]`**       | 默认 [0,0.6,0.85,1,1]；建议 \varphi(1)\in[0.5,0.7] 等见计划 §十                                                |
| P'                     | 缩放后池                        | P' = P_O \times \varphi(L_B)；\varphi=0 则不发转化                                                         |
| T                      | 归因回溯窗口                      | `**attributionWindowDays`** 默认 **7** 天（下单前点赞等）                                                       |
| 决策权重                   | 时间×停留×匹配×内容价值               | `conversion-bonus-service` 四维相乘；`**neutralizeContentQualityInConversionWeight`** 默认真 → 内容价值维恒为 **1** |
| S_r                    | 合并到主评后的「触点分」                | 当前实现为**合并后决策权重和**（同用户多赞取合并逻辑见代码），再参与分配                                                               |
| \Theta                 | 单条评价单订单名义份额上限               | `**thetaCap`** 默认 **0.65** → 单条 \texttt{alloc}_r \le \Theta \cdot P'                                 |
| 二次分配                   | 池须分满                        | `allocateConversionPoolByTheta`：未触顶项按 S_r 迭代分完 P'                                                    |
| L_A                    | 作者可信度等级                     | 同 `getUserTrustLevel`                                                                                |
| \psi(L_A)              | 作者等级对**分成结果**缩放 `**psi[]`** | 默认 [0,0.75,1,1,1.05]；\psi>1 须财务确认                                                                    |
| \texttt{pay}_{r,O}     | 作者实发（税前）                    | \texttt{pay}_{r,O} = \psi(L_A) \times \texttt{alloc}_r（先 alloc 凑满 P'，再乘 \psi）                        |
| `compliancePreTaxOnly` | 转化 pending 是否不做 800 档代扣     | 默认 **true** → `tax_deducted=0`，`amount` 即税前应付                                                        |


**与事后验证**：同一 C 上 **转化 + 事后验证** 共用 \alpha C；互斥顺序以结算任务为准（先转化相关 pending，再事后验证补发等）。

### A.4 单笔订单总硬帽（(B) 双保险）


| 参数                             | 含义                     | 默认                                                                                              |
| ------------------------------ | ---------------------- | ----------------------------------------------------------------------------------------------- |
| `maxUserRewardPctOfCommission` | 上述奖励类型合计相对 C 上限        | **0.8**                                                                                         |
| `reward_source_order_id`       | 交易行上锚定的佣金来源 `order_id` | 迁移 `migration-20260415-transactions-reward-source-order.sql` 后写入；汇总见 `order-reward-cap-service` |


纳入合计的类型：`rebate`、`like_bonus`、`upgrade_diff`（**仅当运营开启差额结算且实际入账**）、`conversion_bonus`、`post_verify_bonus`（**税前负担** = `amount` + `tax_deducted`），并含未结算的 `conversion_bonus` / `upgrade_diff` pending。

**互动轨特别说明**：`like_bonus` **仍计入**上表汇总（转化侧「已发常规点赞」扣减、总支出审计）；但 **`settleLikeBonus` 入账前不再**调用 `clampPayoutToOrderHardCap`，故单订单合计可能因全站池分配结果**高于** 0.8·C，属现行产品设计。

**废止口径**：历史上「基础 + 追评差额 + 常规点赞追加 ≤ 实收佣金 80%」的拆分表述，与**已取消的「追评单独追加奖励」**一并废止；**转化 / 升级差额等路径**仍可能按 `order-reward-cap-service` 在发放时压减；现行以本表所列 `transactions` 类型按单汇总，默认上限 `**maxUserRewardPctOfCommission`·C（80%）**。

### A.5 数值算例（虚构，便于手算核对）

**设定**：某单计费金额 M=10000 元，佣金率 r=6，故 **C=600** 元。平台默认：`complianceRedLine=60`（用于基础三重封顶）、`baseInteractionCapPct=60`、`interactionPoolShare=0.1`（互动全站池）、`maxUserRewardPctOfCommission=0.8`、`conversionPoolShare=0.1`（每单转化池）、`thetaCap=0.65`、买家等级 3 级 \varphi=1、作者等级 2 级 \psi=1。

1. **基础轨**
  - 设该单复杂度对应 `baseFixed=30` 元，`vehicleCoeff=1.0`，复杂度封顶与佣金封顶均不更紧，则 B_{\text{raw}}=30，B=30。  
  - **无** `premiumFloat` 时，首评税前应发基础部分为 **30** 元（再经 0 级拦截、硬帽等，此处从略）。

2. **互动轨（全站池，叙事）**

- 若该月全站仅本例一单完工，则 **ΣC=600**，`pool = 600 × 0.1 = 60` 元，按当月各主评 `weightSum` 占 **ΣW** 比例分配（**不**再按本单 C×0.5%×权重试算）。  
- `baseInteractionCapPct=60` 与 **基础轨** `maxByCommission` 叙事对齐；**不**等于互动池公式（互动池以 **ΣC** 为底）。

3. **转化轨（简化一条归因订单、两条评价分池）**
  - P_O = C \times \alpha = 600 \times 0.1 = 60 元；\varphi=1 \Rightarrow P'=60。  
  - 两条主评合并权重 S_1=80,\ S_2=70，W=150。一次比例：\text{raw}_1=60\times 80/150=32，\text{raw}_2=28。\Theta P'=0.65\times 60=39，均未触顶，**名义份额** \texttt{alloc}_1=32,\ \texttt{alloc}_2=28（和为 60）。  
  - 作者 \psi=1，故 **作者1 税前 32 元，作者2 税前 28 元**。

4. **总硬帽 0.8C**
  - 0.8C = 480 元。已发与待发各类 `amount+tax` 之和不得超过 **480**；超限则按 `order-reward-cap-service.clampPayoutToOrderHardCap` 压减后发。
