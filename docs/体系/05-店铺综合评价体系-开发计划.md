## 〇、已确认事项（用户澄清）

| 事项 | 结论 |
|------|------|
| 4S 店品牌授权 | 4S 店必须有**主机厂品牌授权书**（最重要），此外还有维修经营备案（一类/二类）；`certifications` 需支持主机厂授权书类型 |
| 系统核验违规 | 见《评价内容设置规范》二、系统自动核验模块；**结算金额与报价偏差超 10% 且无用户二次确认 → 直接认定违规**，扣 20 分/单 |
| 重大违规判定 | 服务商不一致（接单/施工/结算主体不一致）扣 50 分/单，累计 2 次直接清退；其他核验违规 10/20 分/单 |
| 店铺下架 | 2 次及以上重大违规 → 得分清零、**shops.status=0** |

## 二、数据源与依赖分析

### 2.1 已有数据源

| 项目 | 表/字段 | 说明 |
|------|---------|------|
| 持证技师 | `shops.technician_certs` | JSON `[{level,years,...}]`，level 含 中级工/高级工/技师/高级技师 |
| 品牌授权 | `shops.certifications` | JSON `[{type,name,image}]`，当前含 license、qualification_cert；**扩展** type=`oem_4s`（主机厂品牌授权书）、`parts_brand`（配件品牌授权） |
| 维修时效 | `orders.duration`、`orders.completed_at`、`orders.accepted_at` | 可计算实际工期 vs 承诺工期 |
| 配件合规 | 无 | 需新增订单级/店铺级记录 |
| 重大违规 | `shop_violations` | 当前仅 progress_not_synced(5)、parts_not_shown(15)，**需扩展** penalty=50 类型 |

### 2.2 依赖关系

- **材料完整率**：不做。1→2 必须材料完整才能提交，无需单独统计。
- **配件合规匹配率**：按**备案制**实现，见阶段 4。
- **重大违规**：依赖系统核验违规（无报价确认、额外项目、服务商不一致、结算偏差等）的判定与写入

## 三、开发计划（按优先级）

### 阶段 1：补齐固定加分项（低依赖，可立即开发）

**目标**：持证技师、品牌授权两项固定加分接入 `calcHardBonus`。

| 序号 | 工作项 | 说明 | 预估 |
|------|--------|------|------|
| 1.1 | 持证技师加分 | 解析 `technician_certs`，高级技师/技师 +8，中级工/初级工 +3，最高 +20 | 0.5d |
| 1.2 | 品牌授权加分 | 扩展 `certifications` 支持 type=`oem_4s`（主机厂品牌授权书）、`parts_brand`（配件品牌授权）；主机厂 4S +15，配件品牌 +5 | 0.5d |
| 1.3 | 文档同步 | 更新《数据库设计文档》《M08-维修厂信息》certifications 类型说明 | 0.5h |

**实现要点**：
- 持证技师：`level` 含「高级技师」「技师」→ +8；「中级工」「初级工」→ +3；封顶 20（见 `utils/shop-profile-constants.js` TECHNICIAN_LEVEL_OPTIONS）
- 品牌授权：4S 店必须有**主机厂品牌授权书**（最重要）；`certifications` 新增 type：`oem_4s`（主机厂 4S 授权书）、`parts_brand`（配件品牌授权）；服务商补充信息页需支持上传

### 阶段 2：维修时效履约率（中依赖）

**目标**：按本季度完成订单计算时效履约率，写入店铺或实时计算。

| 序号 | 工作项 | 说明 | 预估 |
|------|--------|------|------|
| 2.1 | 履约率计算逻辑 | 本季度完成订单中，实际工期 ≤ 承诺工期（duration）的占比 | 0.5d |
| 2.2 | 数据存储 | 方案 A：`shops.timeliness_rate` 定时更新；方案 B：`calcHardBonus` 内实时查询 | 0.5d |
| 2.3 | 集成 calcHardBonus | 100% +5，<70% -20 | 0.5h |
| 2.4 | 定时任务/触发点 | 订单完成时或定时任务更新 shops 字段（若选方案 A） | 0.5d |

**实现要点**：
- 实际工期 = `completed_at - accepted_at`（或接单时间）
- 承诺工期 = `orders.duration`（天）
- 需确认 `accepted_at` 是否存在；若无则用 `orders.created_at` 近似

### 阶段 3：重大违规记录（一票否决）

**目标**：支持 penalty=50 的重大违规，2 次及以上得分清零、店铺下架（status=0）。

**系统核验违规扣分表**（《评价内容设置规范》二、系统自动核验模块）：

| 核验事项 | 违规判定 | 扣分 | 类型 |
|----------|----------|------|------|
| 无用户确认报价 | 无用户线上确认的正式报价单 | 10 分/单 | 普通 |
| 未经确认的额外项目/费用 | 新增项目/费用无用户二次确认 | 20 分/单 | 普通 |
| 结算金额与报价一致 | 偏差超 10% 且无用户二次确认 | 20 分/单 | 普通 |
| 服务商一致 | 接单/施工/结算主体不一致 | **50 分/单** | **重大**，累计 2 次清退 |

| 序号 | 工作项 | 说明 | 预估 |
|------|--------|------|------|
| 3.1 | 扩展 shop_violations | 支持 `violation_type`：`no_quote_confirm`(10)、`extra_project`(20)、`settlement_deviation`(20)、`service_mismatch`(50)；penalty 存扣分值 | 0.5d |
| 3.2 | 系统核验写入入口 | 订单完成/评价页加载时核验，违规则写入 shop_violations；结算偏差超 10% 见 server.js 1816 行 orderVerification.settlement_match | 1d |
| 3.3 | calcHardBonus 重大违规逻辑 | 统计 penalty=50 的次数：1 次 -50；2 次+ 得分清零 | 0.5d |
| 3.4 | 店铺下架逻辑 | 2 次+ 重大违规时 `shops.status=0`，列表/竞价过滤已按 status 过滤 | 0.5d |
| 3.5 | 文档同步 | 更新《商户合规与申诉-实施计划》《05-店铺综合评价体系》3.1 单笔违规扣分 | 0.5h |

**实现要点**：
- 普通违规（10/20 分）计入季度合规率；重大违规（50 分）单独计数，影响 calcHardBonus 与下架
- 下架：`shops.status=0`（已确认）

### 阶段 4：配件合规匹配率（备案制）

**目标**：100% +10，1 次不合规 -20。按**备案制**实现，难度小。

**备案制逻辑**：
- 服务商提交含配件编号照片的材料 → 默认合规（平台不做自动核验）
- 车主事后有疑问可自行查询；查出不合规可通过投诉或追评反馈
- 人工确认服务商配件不合规后 → 写入违规记录，影响店铺配件合规率

| 序号 | 工作项 | 说明 | 预估 |
|------|--------|------|------|
| 4.1 | 违规记录扩展 | 新增 `violation_type`=`parts_non_compliant`，penalty 影响配件合规统计 | 0.5d |
| 4.2 | 店铺级统计 | 本季度完成订单中，无配件不合规记录的占比；1 次不合规 -20，100% +10 | 0.5d |
| 4.3 | 集成 calcHardBonus | 100% +10，有 1 次不合规 -20 | 0.5h |
| 4.4 | **前端提示（必做）** | 服务商提交材料时、车主查看完工材料时，均提示：**「材料通过不代表平台认可配件合规，车主可自行查询配件编号」** | 0.5d |

**实现要点**：人工确认配件不合规的入口（投诉处理/追评人工复核）需与客服流程打通。

## 四、建议实施顺序

| 优先级 | 阶段 | 理由 |
|--------|------|------|
| P0 | 阶段 1（持证技师、品牌授权） | 无依赖，数据已有，改动小 |
| P1 | 阶段 3（重大违规） | 一票否决，合规底线，优先 |
| P2 | 阶段 2（维修时效履约率） | 数据可算，逻辑清晰 |
| P3 | 阶段 5（星级展示） | 完善用户体验 |
| P4 | 阶段 4（配件合规备案制） | 备案制+前端提示，难度小 |
