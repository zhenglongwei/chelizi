# 全链路激励驱动体系

> **定位**：驱动用户持续产出真实、高价值评价，完全服务于「保障评价真实性」的核心目标，而非拉新补贴。评价**现金**已**平台化**为**基础奖金 + 互动奖金 + 转化奖金**三轨，默认参数与公式见《评价与点赞奖励-定期结算方案》**附录 A** 及运营后台 `reward_rules.platformIncentiveV1`（与 `web/api-server/constants/platform-reward-v1.js` 合并）。**追加评价、可选文字、展示与风控**见《02-评价内容质量等级体系.md》，**不在本文件**展开为单独现金名目。

---

## 一、设计原则

- **三轨分工**：基础轨（首评主评、客观完成度 + 订单维度 + 封顶红线）/ 互动轨（点赞与规划中的阅读质量月结）/ 转化轨（内容转化 + 事后验证共帽月结），硬帽与税前口径见附录 A
- **奖励仅与完整性、真实性绑定**：与评价正负倾向（好评/差评）完全无关，符合内容要求即全额发放
- **单订单基础封顶（默认 60%·C）与全类型硬帽（默认 80%·C）**：前者对应根级 `complianceRedLine` / `baseInteractionCapPct` 叙事（**仅基础轨**三重封顶中的佣金项）；**互动轨**为当月完工订单 **ΣC × interactionPoolShare**（默认 10%）全站池，见附录 A **A.2**；后者为 `maxUserRewardPctOfCommission`（附录 A **A.4**），统计类型见 `order-reward-cap-service`（**不含**代理人 `referral_l1_bonus` / `referral_l2_bonus`）。**「订单实收佣金」**与实现侧 `computeOrderCommissionAmount` 一致，**非**劳务税/企业所得税口径
- **分销客源买家**：经推荐绑定用户（`is_distribution_buyer`）成交时，**转化轨**与**事后验证补发**对评价作者的分配**减半**；**互动轨**不变。详见《04》与《辙见需求文档》1.3.4。
- **80% 客观可校验**：评价体系以客观内容为核心，20% 主观为补充，不搞「主观好评换奖励」
- **全流程资金闭环**：通过持牌第三方支付分账，平台不触碰维修资金、不形成资金池。**佣金实收**：当前实现补充支持服务商通过微信支付向平台缴纳佣金及预充值（平台账本 + 微信商户收款），与「用户维修款不经平台」并行；中长期可接入持牌分账与维修款过台方案，二者在需求文档「1.3.1 佣金收取」中已分层说明。

### 2.1 首次评价奖金（基础轨）

```
基础奖励 = 复杂度基础固定奖励 × 车价校准系数
```

**方案 A**：维修复杂度校准系数已并入基础固定奖励，公式简化，避免同一维度重复乘算。


| 维修复杂度 | 普通项目  | 保险事故车 |
| ----- | ----- | ----- |
| L1 极低 | 10 元  | 20 元  |
| L2 低  | 30 元  | 60 元  |
| L3 中  | 150 元 | 300 元 |
| L4 高  | 450 元 | 900 元 |


注：L1 评价为核心定位优化，不发现金奖励，仅保留评价入库与权重计算。

**车价校准系数**：5 档可配置（运营后台模块2），默认 10 万及以下 1.0；10–20 万 1.2；20–30 万 1.5；30–50 万 2.0；50 万以上 3.0。优先用具体价格查表：有 vehicle_price 用裸车价；无则用大模型返回的 vehicle_price_range（官方指导价区间万元）取上限转为 vehicle_price_max（元）查表；无 vehicle_price_max 时回退到 vehicle_price_tier（low/mid/high）→ low=1.0、mid=1.3、high=1.5；或按品牌兜底（沃尔沃/奔驰/宝马等→1.3）。

**维修复杂度升级**：L1/L2 项目数 > 3 时升级 1 级（L2→L3、L1→L2）。

**复杂度认定与回写口径（与实现一致）**：

- **以当前有效的维修项目明细为准**：竞价阶段按该条报价的明细行（含 `damage_part` / `name` / `item` 等可匹配字段）与运营后台 `reward_rules.complexityLevels` 关键词做匹配；订单生成后按订单内**当前方案** `repair_plan.items`（含双阶段报价锁价后的快照）匹配。与定损 AI 单独输出的复杂度标签脱钩；定损文本仅在归一化项目时作辅助上下文。
- **随方案与金额刷新**：服务商修改维修方案、提交或确认到店最终报价等导致明细或有效报价金额变化时，系统会重算并回写订单上的复杂度、奖励金预览与佣金率等字段。车主端竞价列表中对每条报价展示的复杂度与奖励为**只读预估**，以选厂锁定及后续确认方案为准。
- **关键词运营建议**：`complexityLevels` 中 L3/L4 应覆盖常见高危场景用语（如发动机相关、变速箱、**泡水/水淹/进水**、事故结构件等）。存量库若仅有「泡水」而无「水淹」，建议在后台规则中增补同义词，避免明细文案与配置不一致时落回默认 L2。

**维修订单佣金比例**（用于计算实收佣金、60% 基础红线；**固定费率**，无金额分档、无合规率上下调、无价格竞争力乘数）：

- 配置位置：运营后台「奖励金规则」模块 5，`reward_rules` 内 `**commissionRepair`**。  
- **第一维：付款方** — `self_pay`（`is_insurance_accident=0`）与 `insurance`（`is_insurance_accident=1`）各有一个 **default（%）**，默认可配置为如 **6% / 12%**（以线上配置为准）。  
- **第二维：项目类目（可选）** — 在 `byCategory` 中按类目名覆盖对应付款方费率；类目由报价行关键词解析（与 `repair-commission-category` 及搜索类目一致），**未命中类目时用该付款方 default**。  
- **标品**：仍使用系统配置 `product_order_platform_fee_rate`（**后台入口**：Web 管理后台「佣金规则配置」`/admin/commission-rules`），**不**写入 `commissionRepair`。

**基础轨**首评应发仍满足 **≤ 该订单实收佣金 ×（complianceRedLine，默认 60%）**（与 `reward-calculator` 中 `min(baseReward×车价系数, …)` 的佣金项一致）；费率调低时封顶随之降低。商户侧合规问题通过处罚、限流等非佣金手段处理，不通过佣金公式浮动。

**口径说明（与代码、合同披露一致）**：**订单实收佣金** = 当前订单用于计费的有效金额（与 `reward-calculator` / 订单字段一致）× 约定平台佣金费率所得金额（元）。**三轨现金**在系统中按**居民个人劳务报酬**规则作**预扣预缴**（减除与预扣率表二见《评价与点赞奖励-定期结算方案》及 `web/api-server/utils/labor-remuneration-withhold.js`），**代扣发生时点**为**本笔奖励金入账**（`transactions` 计增、`users.balance` 增加、或等效的待回溯/待结算应发确认），**非**用户发起**提现**之时；`transactions.amount` 为**税后实发**列，`tax_deducted` 为同笔代扣。年度汇算等由**税务/支付/合同**侧另行安排。**基础轨**佣金项封顶：`complianceRedLine` 默认 **60%**。**互动轨**为全站月度池，见附录 A，**不**用单笔 `complianceRedLine` 公式计发。

**订单分级（仅阈值）**：`reward_rules` 中 `orderTier1Max` / `orderTier2Max` / `orderTier3Max` 仅用于订单金额分档（展示、统计、`getOrderTier` 等）。**基础轨现金**不再使用「按订单分级的单订单封顶」「按复杂度的单订单封顶」、车价封顶上浮与浮动校准、单项目封顶等；实现以 `web/api-server/reward-calculator.js` 为准：**基础奖励 = `baseReward[L]`（或保险档 `baseRewardInsurance[L]`）× 车价系数，再与实收佣金 × `complianceRedLine%` 取 min**。`complexityLevels` 表格仅用于**关键词匹配 L1～L4**；配件数「破格升档」已关闭。线上历史 JSON 中若仍含旧封顶/浮动字段，应清零或与后台「保存全部」一致，可执行 `node web/scripts/normalize-reward-rules-db.js`（见 `web/deploy.md`）。

**首评发放**：不设「选答 10% 加成」；**不设**据补充说明/AI 内容档位的 `**premiumFloat` 现金加项**（与附录 A **A.1** 一致）。**（v2）**完成全部必答客观题且主观达有效标准后，按规则计发基础轨。**（v3 极简·当前）**完成 §2.0 所述客观路径即可参与基础轨；主观文本为选填。首评主评入账金额以附录 A **A.1** 的基础实发 **B**（经车价系数、佣金红线、账号等级等）为准，**实现细节以 `reward-calculator` 为准**。**0 级**账号不发首评基础轨；**互动/转化**见附录 A、《04》与《01》。

**极简表单 v3**：**五星序**客观题（含 **流程透明度** `process_transparency_star`）+ 场景；**不要求**「主观 ≥5 字」作为领基础奖前提（与 v2 区分）。仍受反作弊/安全审核、账号等级与佣金封顶等约束。**历史客户端**仅提交四维 `five_star` 且无流程维时，有效性与基础轨判定以《02》历史兼容条为准。**过程—评价吻合系数**与「评价—过程证据极端冲突」队列 **pending** 均**不作为**基础轨扣发、缓发条件（与《05》店铺分权重链路区分）。

**与《02》对齐**：评价是否有效、客观题与可选补充说明见《02》；**本文件**只约定三轨现金与红线。

### 2.2 报价证据链与评价展示（双阶段报价）

- 主评价页 **固定展示** **预报价**与**最终报价**摘要（及关键差异），与订单快照字段一致，降低「只看星级不懂价差」的信息不对称。
- **系统核验**、防刷与可校验数据为主；**预↔终** 异常偏离可与店铺「报价准确度」硬指标、合规扣分联动（详见《05-店铺综合评价体系》、[《评价内容设置规范》（已归档）](../已归档/评价内容设置规范.md)）。
- **证据链加权/降权**的具体系数与边界在落地时须与《02》《01》一致，避免重复惩罚或冲突；未上线新系数前，以现有核验项与列表策略为准。

## 三、长效激励：点赞追加奖金

**核心逻辑**：锚定评价对其他车主的真实决策价值，实现「内容价值越高，收益越高，上不封顶」。

详见 **[04-点赞追加奖金体系](./04-点赞追加奖金体系.md)**，包含：

- **常规点赞追加**：基于内容认可度的基础追加
- **内容转化追加**：基于内容决策价值的超额追加
- **事后验证有效点赞**：修车后回头点赞的专属激励

**结算周期**：点赞追加奖金（含常规、内容转化、事后验证）每月 10 日定期结算，结算上月应得；到账预留 1～3 个工作日复核。用户可查看详细清单。

## 五、合规红线（不可突破）

1. 奖励金仅与评价完整性、真实性绑定，与好评/差评无关
2. **子帽**：**基础轨**相对 **C** 的默认红线为 **60%**（`complianceRedLine`）；**互动轨**为 **ΣC×10%** 全站池（默认 `interactionPoolShare`）；**全类型合计**另受 `**maxUserRewardPctOfCommission`** 统计与转化发放等约束，默认 **80%·C**（附录 A **A.4**、`order-reward-cap-service`）；`like_bonus` 入账不经过互动向的按单 clamp
3. 评价 80% 客观可校验，不搞「主观好评换奖励」
4. 全流程资金通过持牌支付分账闭环
