# 评价内容质量体系

> **定位**：保障评价**可校验、可排序、可风控**，杜绝水评与虚假评价；与《用户账号可信度等级体系》并列为底层支撑。**产品口径**：**不再使用「评价内容 1～4 级」（基础/优质/标杆/爆款）」作为对车主或运营的叙事**；车主侧理解与宣导应为 **「客观题各维打分领基础奖 → 互动情况 → 转化情况」**（现金细则见《03-全链路激励驱动体系》与《评价与点赞奖励-定期结算方案》附录 A）。
>
> **权责拆分**：是否**有效**、客观题完成度**仅与车主提交的表单**相关；结算单、完工图、维修过程材料为**服务商义务**，材料缺失扣服务商分，**不**影响车主基础轨资格（仍须账号、反作弊与佣金硬帽等，见《03》）。
>
> **实现侧说明**：库表与接口可能仍保留 `content_quality`、`content_quality_level` 等字段供**后台审核、历史兼容与审计**；其与首评**基础奖金**无默认绑定，**不得**对用户展示或承诺「升到某级多拿钱」。

## 一、核心定义


| 类型       | 定义                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               | 权益                                                        |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| **有效评价** | **（表单 v2）完成场景内全部必答客观题（自费 5 题 / 事故车 6 题）+ 主观描述达标（≥5 字且非纯水评）。（表单 v3 极简·当前）完成 `objective_answers.version=3` 且 `v3_form=five_star` 路径下**五项** 1～5 星：**流程透明度**（`process_transparency_star`，1=认为流程完全不透明，5=完全透明）、报价透明度（`quote_transparency_star`）、配件与材料可溯源度（`parts_traceability_star`）、故障整体修复程度（`repair_effect_star`，含外观认知合并）、服务态度（`service_experience_star`）；可选 `owner_verify_result`；主观补充选填**（≤200 字，可不填），有内容时须过反作弊/安全审核、不得纯水评。**（v3 历史兼容·仅四维星）**在库内无 `process_transparency_star` 或该键无效时，仍可将「报价、配件、修复、服务」四维齐视为有效 v3（与上线前客户端一致）。**（v3 历史兼容·旧表单）**仍支持仅 `parts_authenticity_check` 三选一 + `repair_effect_star` + `service_experience_star` 或 `fault_fix_effect` + `overall_star` 组合视为有效 v3 | **（v3）客观题路径完成即可参与基础轨**（见《03》）；计入店铺加权；对外展示。**（v2）**另须主观达标。 |
| **无效评价** | 未完成必答客观题；纯水评；重复/AI 生成；高风险账号；恶意诋毁/广告等，按规则或审核判定；**经《05》「评价—过程证据极端冲突」队列结案并勾选「评价无效」**时与上列无效路径一致（`content_quality=invalid` 等）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         | 不计入店铺评分，不发放奖励金，按规则仅商户可见或待人工裁定                             |


**补充内容（非「等级」）**：车主可选填文字、上传凭证；用于**列表摘要、验真展开、风控抽样**等，**不**作为领取基础轨的前置条件（v3）。**首评现金已取消**按「内容优质」叠加的 `premiumFloat`；千问等返回的档位/标签仅用于**展示、风控与无效判定**，**不**与基础轨金额挂钩。

## 二、评价有效性校验（仅车主输入）

**原则**：是否有效**仅看车主提交的客观题（及各维打分）与必选校验**；不依赖服务商上传材料。

### 2.1 必答客观题（自费 5 题 / 事故车 6 题）

提交页按 [《重构评价体系》第二节（已归档）](../已归档/重构评价体系.md)：**自费维保**为 `q_pay_match`、`q_process_transparent`、`q_parts_match_quote`、`q_fault_resolved`、`q_warranty_provided`；选「金额不一致」时须填 `q_pay_mismatch_note`（≤10 字）。**事故车**另含 `q_no_fraud_inducement`、`q_loss_explained`、`q_repair_matches_loss`、`q_insurance_smooth` 及共用的故障/质保两题。`reviews.objective_answers` 存 JSON（含 `scene`、`version: 2` 及上述键）。**过渡**：服务端可将历史三题 `q1_progress_synced` / `q2_parts_shown` / `q3_fault_resolved` 映射为过程透明、配件一致、故障解决，但缺新题仍视为未完成。

**星级联动**：客观题选「否」时，对应体验维度星级服务端封顶（最高 2 星），详见 `web/api-server/utils/review-objective-schema.js`。

**C 端列表/详情**：展示用语可与提交页题干对齐为短标题 + 说明；历史数据仍可能仅为旧三题字段，列表侧需兼容展示。**店铺详情 / 口碑流**对单条评价默认「摘要 + 展开验真」；对外接口返回的报价历程**不含举证原图 URL**，`quote_snapshot` **不含分项行价**（保留轮次总价等）；若车主主评价选择不公示图片（`reviews.review_images_public=0`），公示接口对**店端留档/维修前后等**仍按 `review_public_media` 过滤；**车主在本页自主上传**的图片 URL 写入 `objective_answers.owner_always_public_urls` 的，**仍参与公示合并**（与店端素材授权解耦）。私域（本人订单详情等）仍完整展示。与《04-维修厂详情页》一致。

#### 2.1.1 极简主评价表单 v3（`review_form_version=3`，与 v2 并存）

- **客观与星级（当前·五星序）**：`process_transparency_star`、`quote_transparency_star`、`parts_traceability_star`、`repair_effect_star`、`service_experience_star` 各 1～5；`objective_answers` 带 `version: 3`、`v3_form: 'five_star'` 与 `scene`；可选 `owner_verify_result`。**兼容历史**：无 `process_transparency_star` 时仍可以四维 `five_star` 判有效；仍支持仅 `parts_authenticity_check` 三选一 + 双星，或 `fault_fix_effect` + `overall_star`。
- **入库映射**：`reviews.rating` = `service_experience_star`（无则回退 `overall_star`）；`ratings_quality` = `repair_effect_star`；`ratings_price` = `quote_transparency_star`（无则与 `rating` 一致）；`ratings_parts` = `parts_traceability_star`（无则与 `rating` 一致）；`ratings_service` / `ratings_speed` 与 `rating` 对齐写入（见 `review-service-minimal-v3.js`）。`**process_transparency_star` 仅存于 `objective_answers` JSON**，不单独映射到 `reviews.ratings_*` 列，供过程留痕/AI 吻合与运营分析使用。
- **公示素材**：`reviews.review_public_media` 分项勾选仅约束**店端/订单侧留档图**；`objective_answers.owner_always_public_urls` 所列本页车主上传 URL **强制并入**公示接口下发的实拍类列表（实现见 `review-public-system-sanitize.js` 的 `applyGranularPublicImages`）。`reviews.review_system_checks` **库内存完整 JSON**（含报价偏离归纳 `quote_deviation`、外观 AI `appearance`、配件侧 `parts_delivery` 内 `ai_match_level` / `coverage_summary` 等，供后台与矛盾检测）；**车主端与公示接口**仅通过 `sanitizeSystemChecksForUserFacing`（与历史函数名 `sanitizeSystemChecksForPublic` 同义）下发**订单留痕子集**：`quote_flow`（节点与 disclaimer/chart_spec）、`parts_delivery` 中仅 `**merchant_methods` / `merchant_verify_note` / `status`**（若有）、以及 `warranty` / `loss_vs_settlement` 等占位事实；**不下发**外观整块、`quote_deviation`、配件归纳文案与一切 `ai_*` 结论字段。**配件与方案一致性**的完整推理仍见 `orders.completion_evidence.parts_traceability_ai`。
- **分歧留痕**：`review_system_checks.user_ai_alignment.has_divergence` 可由**任一新维星 ≤2**（含 `process_transparency_star`）、`parts_authenticity_check=mismatch` 或（若仍提交）完整 `ai_user_alignment` 覆盖项触发，供管理端 `pool=ai_divergence` 抽样；**不**与基础轨发奖自动挂钩。
- **星级 vs 系统/AI 自动矛盾**：主评价极简 v3 **提交时**按当时 `review_system_checks` 写入 `star_ai_anomaly`（仅当存在反差且后台开关开启）：对比 `quote_transparency_star` / `repair_effect_star` / `parts_traceability_star` 与当单库内 `quote_deviation`、`appearance`、`parts_delivery` 的正负向归纳；如「报价偏离归纳为低但车主报价维 ≤2★」或「配件 AI 不匹配但配件维 ≥4★」等。（`process_transparency_star` 与过程 AI 吻合规则见产品「完工不卡审与过程 AI 评信」方案，**不**默认写入 `star_ai_anomaly` 现有三轴。）若外观分析为 **提交后异步** 写入（`appearance.status` 由 pending 变为有结果），服务端在 **异步 UPDATE `review_system_checks` 成功后** 会**再算一次** `star_ai_anomaly` 并写回，避免漏标。含 `flag:1` 供 SQL 筛 `pool=star_ai_anomaly`；**不下发**车主端/公示（`sanitizeSystemChecksForUserFacing` 不包含该键）。**阈值可配置**：`settings` 键 `review_star_ai_anomaly_enabled`、`review_star_ai_anomaly_user_low_max`、`review_star_ai_anomaly_user_high_min`、`review_star_ai_anomaly_quote_pct_good_max`、`review_star_ai_anomaly_quote_pct_bad_min`、`review_star_ai_anomaly_repair_good_min`、`review_star_ai_anomaly_repair_bad_max`（缺省与代码内 `review-star-ai-anomaly-config.js` 的 `DEFAULTS` 一致）；管理端「评价审核」页「星级-AI 矛盾阈值」保存至上述键。计算逻辑见 `web/api-server/utils/review-star-ai-anomaly.js`。

### 2.2 主观描述（可选；一票否决仅针对「有内容」）


| 校验项   | 硬性标准                                                                    |
| ----- | ----------------------------------------------------------------------- |
| 内容有效性 | **（v2）**至少 1 句与维修项目相关描述；纯水评无效。**（v3）**可不填；若填写则须 ≤200 字；**不**以「过短/不优质」等作提交拦截；有文字时做与历史主评的**相似度**等轻量反作弊，及千问在**有服务商材料图**时的**图-文明确矛盾/底线合规**（见 §2.5） |
| 账号合规性 | 按《01》用户可信度等通用规则，不因注册时长单独判定无效                                            |
| 行为合规性 | 非重复内容、非 AI 生成、非恶意诋毁/广告；无图仅文字辱骂的恶意差评无效                                   |


### 2.3 无效评价判定

出现以下任一情况可判无效：未完成场景内全部必答客观题；纯水评；内容重复、AI 生成；发布账号高风险；恶意诋毁/广告。

**引导式提交**：不强制车主完成必答题或最低字数，采用引导方式。若提交不满足有效标准，提示改进建议与潜在奖励；车主可选择「仍要提交」强行提交：**规则校验 / 客观题 / 反作弊**等路径仍可能入库为**无效评价**（`content_quality=invalid` 等，仅商户可见、不计入得分、不发放奖励金）。**待运营人工裁定（`pending_human`）**（首评/整体性重评等视路径）：**整体性重评**仍可能走原全量 `analyzeReviewWithQwen` 的 `pass`；**主评价首提**因材料/表述矛盾等 `force_submit` 仍可能进入人工的，以 **v2 旧路径 + storeAiRejectedPendingHumanReview** 为准（极简 v3 若未接该入库分支则仅 400 提示，见实现）。**不因**仅本地「字数/水词」规则单独转 `pending_human`。原要求千问在 `pass=false` 时须在返回 JSON 的 `**details.operationsReview`** 中给出 `**summary`（运营摘要）** 与 `**notMetItems`（未达标要点列表，至少 1 条）**，与面向车主的 `**rejectReason`** 区分；服务端在模型漏填时从合规等字段回填。运营后台「评价审核」列表与「AI 驳回裁定」弹窗展示上述字段，供审核员裁定**有效**（按规则发奖并展示）或**无效**（维持不展示不发奖）。

### 2.4 材料与服务商义务

结算单、完工图、维修过程材料为**服务商义务**（completion_evidence）。材料缺失**不影响**评价有效性、车主基础轨资格；材料缺失应扣减服务商合规分，纳入店铺硬指标。详见《05-店铺综合评价体系》。

### 2.5 安全审核、追评与整体性重评（展示 / 风控，非「升级领钱」）

**千问 / 首评提交流程（现行）**：`POST /api/v1/reviews` 在配置千问且本单**存在服务商侧可审图**（车主补充的完工/结算等 URL）时，**仅**调用 `analyzeReviewEvidenceContradictionWithQwen`：判 **服务商留档图** 与 **车主文字+客观题** 是否存在**明确事实矛盾**，以及**底线合规**（明显广告、严重辱骂等）。**不**对文字长短、是否「优质」、是否模板化做拦截。无图时首评不跑该视觉调用，文字侧仅**相似度**等规则（v3 不做人工作最短字数门槛）。`details` 回写 `textImageContradiction`、`contentViolation` 等供审计。  
**千问 / 整体性重评等**：仍可用原 `analyzeReviewWithQwen` 路径做**运营向**整评/摘要（与首评 `pass` 门槛分离）；**不**作为对 C 端用户的「内容 1～4 级」发钱话术。

**整体性评估**：主评价与追评视为**同一评价整体**。用户可**随时追加追评**，不限制时间与次数。追评提交后可触发整体性重评（`recomputeHolisticContentQuality`），更新**展示标签、风控字段、内部审计信息**；**现金差额**（`upgrade_diff`）默认关闭，见《03》§2.2。

**整体性重评的千问入参**：以结构化 JSON 区分 `**firstReview`（首评）** 与 `**followUps`（各次追评，含阶段标签）**，并附 `**auditInstruction`**；**图片**仍取首评所附；**客观题**以首评提交时为准。

**整体性重评结果**：`pass: false` 时回写 `pending_human` 等；`pass: true` 时合并 `ai_analysis`、审计日志。**不对用户承诺**「重评通过即升级内容档位或多发现金」。

### 2.6 追评判定规则（结合质保期等，影响展示与参考价值叙事）

追评对**后续展示、运营解读**的影响需结合**质保期**、**追评内容类型**综合判定：


| 场景           | 判定               | 说明                                              |
| ------------ | ---------------- | ----------------------------------------------- |
| 质保期内         | 正常纳入整体评估         | 可与首评一并用于摘要与验真展示                                 |
| 超质保期 · 故障复发  | **弱化「单次故障复发」叙事** | 质保期后称「修的地方又坏了」，属正常磨损语境时，参考价值有限，**不**自动推导为「降档扣钱」 |
| 超质保期 · 质量依然好 | **可强化正面叙事**      | 质保期后称「维修质量依然很好」，可作口碑佐证                          |
| 配件类型不匹配      | **不受时间限制**       | 称「承诺原厂件实际非原厂」等，若用户提供可证明的凭证，可作为纠纷与展示依据           |


**配件不匹配凭证要求**：刚修好时的配件编号照片 + 追评时的配件编号照片（证明同一款配件）+ 合规渠道查询或证明该配件非原厂件。

**质保期来源**：订单 `repair_plan` / `quotes` 的 `**items[].warranty_months`**（单位：月）；不设整单 `warranty` 业务字段。

**分项质保与评价证据**：分项质保义务主体为**接单维修厂**，以订单内已确认的维修/报价方案 `**items[].warranty_months`** 为约定依据；辙见**不提供**独立电子质保卡、印章类凭证或面向用户的防伪核验流程。评价页「售后质保」区以系统从方案解析的**分项质保约定列表**为主展示；历史订单若存在 `completion_evidence` 中的纸质质保类补充图，仍可作为图片证据展示。客观题 `q_warranty_provided` 对应「是否在订单明细中提供明确分项质保（质保月数）」。数据库中 `orders.platform_warranty_card` 等字段可为历史数据保留，新产品路径不再依赖其生成与用户侧展示。

### 2.7 用户举证（故障未解决选否）

`q_fault_resolved` 选「否」时，用户可**选填**上传故障凭证（fault_evidence_images）。**上传**可用于**维权展示、运营抽样、摘要加权**；首评现金仍以《03》基础轨与 `premiumFloat` 配置为准，**不**对用户表述为「举证即升内容级多拿钱」。

## 三、排序、权重与展示（无「评价内容 1～4 级」产品档）

- **列表与详情排序**：默认综合 **客观维星级/问卷完成度**、**阅读与点赞等互动信号**、时间新鲜度、距离等；店铺维度与全站流策略见《05》《06》及列表配置。
- **现金**：基础轨仅依赖**客观完成度 + 订单规则**（《03》）；**互动轨、转化轨**见《03》附录 A 与《04》。
- **库内字段**：`content_quality_level` 等若仍存在，仅供**兼容与后台**，**不**写入车主可见的「等级」产品文案。

## 四、单条评价权重计算（示意）

```
单条评价总权重 = 订单含金量权重 × 评价有效性与客观信号权重 × 用户账号可信度权重 × 合规特殊系数
```

（互动热度、转化归因等在《04》《06》与结算任务中另计，**不**与本条公式重复乘算同一现金。）


| 维度         | 系数配置（可随策略调整）                                    |
| ---------- | ----------------------------------------------- |
| 订单含金量      | L1: 0.2；L2: 1.0；L3: 3.0；L4: 6.0；保险事故车 ×2        |
| 评价有效性与客观信号 | 无效水评: 0.1；**有效且客观题完成**: 1.0；带有效举证/验真通过等运营认定系数另配 |
| 用户账号可信度    | 由《01-用户账号可信度等级体系》0-4 级决定                        |
| 合规特殊系数     | 正常: 1.0；人工核验通过: 1.2；违规嫌疑待复核: 0.5                |


**说明**：「有效差评」等对权重的上调/保护若仍在 `shop-score` 等实现中使用，以代码与《05》为准，**不**依赖已废止的「评价内容 1～4 级」。

---

## 五、「踩」机制（已废止）

历史上曾规划「踩」作为负向反馈并与点赞互斥。**现行产品已下线踩入口**；**不参与**内容质量等级、单条权重、点赞追加奖金、排序与曝光计算。`review_dislikes` / `reviews.dislike_count` 若库中仍有历史数据，**仅作存档**，新逻辑不读取、不展示踩操作。

## 六、展示规则

- **车主**：各维星级、可选文字、凭证、发布用户等级、**点赞**与阅读；**高互动**、带完整凭证的差评可优先置顶策略由列表配置决定
- **服务商**：评价全量内容、对应订单明细、权重计算明细、申诉入口；差评优先推送
- **评价聚合页**：全平台评价流，按**互动与客观信号、时间**等排序（可切换时间、距离）；每条展示用户昵称、等级、内容、图片、服务项目、金额、时间、店铺（可点击跳转）；**有数据时**展示「报价协商记录（公示）」：默认一行摘要，多条或含证明图可展开明细；新鲜度：展示 1 次后 3 天内不再重复；支持**点赞**、阅读埋点（纳入**互动轨、转化轨**核算，见《04》与附录 A）
