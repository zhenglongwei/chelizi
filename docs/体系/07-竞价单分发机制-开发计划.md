## 〇、文档要点回顾

**原则**：先过滤，后分层，再排序。


| 模块        | 文档要求                                                     |
| --------- | -------------------------------------------------------- |
| **硬门槛**   | 距离、资质、合规率≥80%、近30天无重大违规、同项目完单、产能（近24h无满负荷）               |
| **兜底**    | 过滤后不足 3 家时，按 20% 扩大距离直至凑够                                |
| **综合匹配分** | 店铺综合得分×场景权重 + 同项目完单量分 + 报价历史准确度分 + 时效履约率分                |
| **梯队**    | 第一梯队：≥80分+合规≥95%+无违规；第二梯队：60-79分+合规≥85%；第三梯队：<60分且仅满足硬门槛 |
| **分发数量**  | L1-L2 最多 10 家，有效报价满 5 家停止；L3-L4 最多 15 家，有效报价满 8 家停止      |
| **报价排序**  | 综合匹配分×70% + 报价合理性分×30%；用户可选：价格从低到高、距离从近到远、质保从长到短         |
| **报价展示**  | 完整配件明细、工时费、质保期限、门店资质、历史同项目案例                             |

## 二、数据源与依赖分析

### 2.1 已有数据源


| 项目     | 表/字段                         | 说明                     |
| ------ | ---------------------------- | ---------------------- |
| 店铺综合得分 | `shops.shop_score`           | shop-score 服务计算        |
| 合规率    | `shops.compliance_rate`      | 季度合规率                  |
| 报价偏差率  | `shops.deviation_rate`       | 报价历史准确度                |
| 资质等级   | `shops.qualification_level`  | 一类/二类/三类               |
| 资质审核   | `shops.qualification_status` | 1=通过                   |
| 竞价分发结果 | `bidding_distribution`       | 梯队、匹配分                 |
| 违规记录   | `shop_violations`            | 系统核验违规 penalty=50 为重大  |
| 违规记录   | `violation_records`          | 防刷/违规 target_type=shop |


### 2.2 数据源差异


| 项目        | 说明                                                                                                              | 建议                                                                      |
| --------- | --------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **重大违规**  | 07 文档「近 30 天无重大违规」；bidding-distribution 用 `violation_records` level=4；shop-score 用 `shop_violations` penalty=50 | 统一为 `shop_violations`：近 30 天有 penalty=50 的店铺踢出；与 05 文档一致                |
| **时效履约率** | calcMerchantMatchScore 中 responseScore 固定 5                                                                     | 复用 shop-sort-service 的 `getAvgResponseMinutesByShopIds`，或按 orders 履约率计算 |


### 2.3 依赖关系

- **05 店铺综合评价体系**：已完成，`shop_score` 可用
- **06 店铺列表排序算法**：场景权重、响应速度、距离分等可复用
- **quotes 表**：items 含配件明细与 `warranty_months`；duration 为工期；库内 `warranty` 列为兼容写入（max 分项），接口不展示整单质保

## 三、开发计划（按优先级）

### 阶段 1：报价排序与用户可选排序统一

**目标**：报价排序公式与梯队划分一致，前端展示排序选择器。


| 序号  | 工作项               | 说明                                                                                                                  | 预估   |
| --- | ----------------- | ------------------------------------------------------------------------------------------------------------------- | ---- |
| 1.1 | 统一 calcMatchScore | `bidding-service.sortQuotesByScore` 改为调用 `bidding-distribution.calcMerchantMatchScore`，或传入 bidding 上下文计算与梯队一致的综合匹配分 | 0.5d |
| 1.2 | 前端排序选择器           | `pages/bidding/detail` 增加排序选项：综合推荐、价格从低到高、距离从近到远、质保从长到短；默认「综合推荐」                                                    | 0.5d |
| 1.3 | 文档同步              | 更新《03-竞价报价页》sort_type 说明，与 07 文档对齐                                                                                  | 0.5h |


**实现要点**：

- 报价列表接口已有 `sort_type`：default、price_asc、distance、warranty
- 排序选择器需在报价列表上方展示，切换时重新请求 `getBiddingQuotes(id, { sort_type })`
- 保险支付/自费 与 03 文档的「保险支付优先高报价」可能冲突，需与产品确认：07 文档以用户可选为准

### 阶段 2：重大违规数据源统一

**目标**：硬门槛「近 30 天无重大违规」统一使用 `shop_violations` penalty=50。


| 序号  | 工作项                      | 说明                                                                                    | 预估   |
| --- | ------------------------ | ------------------------------------------------------------------------------------- | ---- |
| 2.1 | 修改 filterShopsForBidding | 将 `violation_records` 查询改为 `shop_violations`：近 30 天有 penalty=50 的 shop_id 踢出          | 0.5d |
| 2.2 | 第一梯队「无违规」                | 文档第一梯队要求「无违规」；当前 assignTiers 仅检查合规率；若需严格「无违规」可增加 shop_violations 近 30 天 penalty>0 的过滤 | 0.5d |
| 2.3 | 文档同步                     | 更新《07-竞价单分发机制》实现参数说明，注明重大违规来源为 shop_violations                                        | 0.5h |


**实现要点**：

- `shop_violations` 表：violation_type 含 service_mismatch(50)、settlement_deviation(20) 等
- 近 30 天：`created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)`
- 重大违规：`penalty = 50`

---

### 阶段 3：综合匹配分时效履约率

**目标**：calcMerchantMatchScore 中时效履约率分使用真实数据。


| 序号  | 工作项                       | 说明                                                                                 | 预估   |
| --- | ------------------------- | ---------------------------------------------------------------------------------- | ---- |
| 3.1 | 响应速度分                     | 复用 `shop-sort-service.getAvgResponseMinutesByShopIds`，或按 orders 履约率（实际工期≤承诺工期占比）计算 | 0.5d |
| 3.2 | 集成 calcMerchantMatchScore | 将 responseScore 从固定 5 改为动态计算（0-100 分制，5 分钟内响应满分）                                   | 0.5d |


**实现要点**：

- 06 文档：响应速度 = 近 30 天平均报价响应时长（竞价创建→店铺首次报价）
- 或：时效履约率 = 本季度完成订单中实际工期≤承诺工期的占比，100% 满分、<70% 0 分

---

### 阶段 4：有效报价满 N 家停止（可选）

**目标**：当有效报价数达到配置阈值时，不再向第二、第三梯队**推送**新消息；已收到推送的服务商在用户选厂前仍可继续报价。

**已确认**（用户澄清）：

- 「停止」= **不再推送**，非「不可见」
- 在用户确认选择报价之前，收到报价推送的服务商一直可以报价


| 序号  | 工作项      | 说明                                                                                       | 预估     |
| --- | -------- | ---------------------------------------------------------------------------------------- | ------ |
| 4.1 | 有效报价判断   | 有效报价 = `quotes.quote_status=0` 且金额>0                                                     | 0h（已有） |
| 4.2 | 第二梯队补发条件 | 若有效报价已达 distributeL1L2ValidStop/distributeL3L4ValidStop，则不向第二梯队补发消息；**已收到推送的店铺仍可见、仍可报价** | 0.5d   |
| 4.3 | 第三梯队补发条件 | 若有效报价已达 distributeL3L4ValidStop，则不向第三梯队补发；**已收到推送的店铺仍可见、仍可报价**                           | 0.5d   |


**实现要点**：

- 仅修改 `sendDelayedBiddingMessagesForShop`、`sendAllDelayedBiddingMessages`：补发前检查有效报价数，未达阈值才推送
- **不修改** `checkTierVisibility`、`listBiddingsForShop`：已收到推送的店铺可见性不变
