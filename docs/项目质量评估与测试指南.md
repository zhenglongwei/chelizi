### 1.2 快速执行（推荐顺序）

```bash
# 1. 算法测试（无需环境，最快）
node web/scripts/test-reward-algorithms.js

# 2. API 完整功能测试（38 用例，默认连服务器）
node web/scripts/test-api-comprehensive.js

# 3. 订阅消息配置校验（24 项）
node web/scripts/test-subscribe-config.js

# 4. 健康检查（服务器）
curl https://simplewin.cn/api/health

# 5. 全流程模拟（默认连服务器）
cd web
API_BASE=https://simplewin.cn node scripts/simulate-full-flow.js

# 6. 后台 E2E（需后台已部署或本地 preview）
cd web && npm run test:e2e

# 7. 竞价分发详情（查看某竞价的邀请名单、梯队、过滤逻辑）
node web/scripts/fetch-bidding-distribution.js BID1234567890
```

### 1.3 竞价分发可见性

竞价创建时会触发 `runBiddingDistribution`，服务端会输出详细日志：

- 过滤通过的店铺列表
- 梯队划分（一/二/三梯队及匹配分、合规率）
- 实际邀请名单与未邀请原因（名额不足）

**API**：`GET /api/v1/admin/biddings/:id/distribution`（需 admin token）返回完整分发详情。

**脚本**：`node web/scripts/fetch-bidding-distribution.js <bidding_id>` 可快速查看。

## 二、安全规范评估

### 2.1 已做得较好的方面


| 项目           | 现状                                                         |
| ------------ | ---------------------------------------------------------- |
| **SQL 注入**   | 使用 `pool.execute(sql, [params])` 参数化查询，未发现直接拼接用户输入到 SQL    |
| **密码存储**     | 使用 `pbkdf2` + 随机 salt，100000 次迭代                           |
| **JWT 认证**   | 用户/服务商接口均有 `authenticateToken` / `authenticateMerchant` 校验 |
| **权限隔离**     | 服务商接口校验 `shop_id`，用户接口校验 `userId`                          |
| **输入白名单**    | 用户 profile 更新、店铺排序等使用字段白名单，避免任意字段写入                        |
| **ORDER BY** | `sort` 参数通过 if-else 映射到固定字符串，无用户输入直接拼接                     |


### 2.2 建议改进项


| 项目                 | 风险                                   | 建议                              |
| ------------------ | ------------------------------------ | ------------------------------- |
| **JWT_SECRET 默认值** | `server.js` 中 `JWT_SECRET            |                                 |
| **CORS**           | `app.use(cors())` 全开放，任意域名可跨域        | 生产环境限制 `origin` 为小程序域名、后台域名     |
| **请求体大小**          | `express.json({ limit: '10mb' })` 较大 | 按业务需要可适当下调，减少 DoS 风险            |
| **无速率限制**          | 登录、注册、短信等接口无限流                       | 引入 `express-rate-limit` 对敏感接口限流 |
| **无 Helmet**       | 未设置安全头（X-Content-Type-Options 等）     | 添加 `helmet` 中间件                 |
| **错误信息泄露**         | 部分 `error.message` 直接返回给前端           | 生产环境统一返回通用错误，详细日志仅写服务端          |


### 2.3 可选安全加固命令

```bash
cd web/api-server
npm install helmet express-rate-limit --save
```

在 `server.js` 中增加（示例）：

```javascript
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');

app.use(helmet({ contentSecurityPolicy: false })); // 若前端有 CSP 可关闭
app.use('/api/v1/auth/login', rateLimit({ windowMs: 15 * 60 * 1000, max: 20 }));
app.use('/api/v1/merchant/login', rateLimit({ windowMs: 15 * 60 * 1000, max: 20 }));
```

## 三、性能与并发

### 3.1 当前配置


| 项目         | 配置                             | 说明           |
| ---------- | ------------------------------ | ------------ |
| **数据库连接池** | `connectionLimit: 10`          | 单进程最多 10 个连接 |
| **请求体**    | `limit: 10mb`                  | 大 JSON/上传场景  |
| **分页**     | `limit` 通常 `Math.min(20, ...)` | 避免单次拉取过多数据   |
| **店铺列表**   | 部分场景 `LIMIT 200` 再内存排序         | 数据量增大时需关注    |


### 3.2 并发支持

- **单进程**：Express 单进程，Node 异步 I/O 可处理较多并发请求
- **水平扩展**：无状态设计，可多实例 + 负载均衡；需注意：
  - MySQL 连接池为每进程独立，多实例会成倍占用连接
  - 定时任务（如结算）若多实例需加分布式锁，避免重复执行

### 3.3 性能建议

1. **索引**：确认 `docs/database/` 与 `web/database/schema.sql` 中高频查询字段有索引
2. **慢查询**：生产环境开启 MySQL 慢查询日志，定期分析
3. **缓存**：热点数据（如店铺列表、用户等级）可考虑 Redis 缓存
4. **PM2 集群**：生产可用 `pm2 start server.js -i 2` 等多实例，注意连接数
