# 竞价流程与状态流转

## 1. 概述

本文档描述从用户提交竞价到竞价结束的完整流程，涵盖所有状态、触发条件及数据变更。

---

## 2. 核心实体与状态定义

### 2.1 竞价 (biddings)


| 字段               | 说明                |
| ---------------- | ----------------- |
| bidding_id       | 竞价唯一 ID           |
| user_id          | 发起人（车主）           |
| report_id        | 定损报告 ID（必填，外键）    |
| status           | 0=进行中 1=已结束 2=已取消 |
| expire_at        | 过期时间（创建时 +2h）     |
| selected_shop_id | 选中的维修厂 ID（成交时写入）  |


### 2.2 报价 (quotes)


| 字段                | 说明                                  |
| ----------------- | ----------------------------------- |
| quote_id          | 报价唯一 ID                             |
| bidding_id        | 所属竞价                                |
| shop_id           | 报价维修厂                               |
| quote_status      | 0=有效 1=成交 2=已失效                     |
| quote_valid_until | 报价有效期截止时间，服务商可自定义，默认 created_at+3 天 |


### 2.3 订单 (orders)


| 字段         | 说明                                                             |
| ---------- | -------------------------------------------------------------- |
| order_id   | 订单唯一 ID                                                        |
| bidding_id | 来源竞价                                                           |
| status     | 0=待接单 1=维修中 2=待验收 3=已完成 4=已取消；**竞价流程**下用户选厂即自动接单，订单直接 status=1 |

## 3. 完整流程与状态流转

### 3.1 流程总览

```
[用户发起竞价] → [竞价进行中] → [结束路径 A/B/C/D]
                    │
                    ├─ 服务商在范围内可见
                    ├─ 服务商提交报价 (quote_status=0)
                    └─ 用户操作：选厂 / 结束 / 过期
```

### 3.2 阶段一：竞价创建


| 步骤  | 触发                       | 数据变更                                         | 备注                   |
| --- | ------------------------ | -------------------------------------------- | -------------------- |
| 1   | 用户完成 AI 定损，点击发起竞价        | INSERT biddings (status=0, expire_at=NOW+2h) | report_id 必填         |
| 2   | 创建时传入 latitude/longitude | UPDATE users SET latitude, longitude         | 用于服务商距离筛选，无则附近服务商看不到 |


**前置条件**：report_id 存在且属于当前用户。

### 3.3 阶段二：竞价进行中 (biddings.status=0)

#### 3.3.1 服务商可见性

服务商能看到某竞价的条件（同时满足）：

1. `biddings.status = 0`
2. `biddings.expire_at > NOW()`
3. `users.latitude IS NOT NULL AND users.longitude IS NOT NULL`（车主有位置）
4. `shops.latitude IS NOT NULL AND shops.longitude IS NOT NULL`（店铺有位置）
5. 距离：`haversine(用户位置, 店铺位置) <= biddings.range_km`

#### 3.3.2 服务商 Tab 分类（同一套筛选逻辑）


| Tab           | 条件                        | 说明          |
| ------------- | ------------------------- | ----------- |
| 待报价 (pending) | 上述可见性 + 本店未报价             | 可进入详情并提交报价  |
| 已报价 (quoted)  | 上述可见性 + 本店已报价             | 可查看自己的报价    |
| 已结束 (ended)   | biddings.status=1 + 本店有报价 | 可查看成交/已失效状态 |


#### 3.3.3 服务商提交报价


| 步骤  | 触发      | 数据变更                                                                                   | 校验                                                            |
| --- | ------- | -------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| 1   | 服务商提交报价 | INSERT quotes（`quote_valid_until = DATE_ADD(NOW(), INTERVAL quote_validity_days DAY)`） | 竞价 status=0；本店未报价；本店在范围内；资质已通过；**竞价窗口**内（`expire_at > NOW()`） |
| 2   | 报价有效期   | quote_validity_days 服务商可自定义，默认 3 天；1–30 天范围内                                           |                                                               |


### 3.4 阶段三：竞价结束（四种路径）

#### 路径 A：用户选择某家报价（成交）


| 步骤  | 触发           | 数据变更                                                                   |
| --- | ------------ | ---------------------------------------------------------------------- |
| 1   | 用户点击「选择」某维修厂 | INSERT orders (status=1 维修中, accepted_at=NOW, repair_plan=报价) **自动接单** |
| 2   |              | UPDATE quotes SET quote_status=1 WHERE 选中店                             |
| 3   |              | UPDATE quotes SET quote_status=2 WHERE 其他店                             |
| 4   |              | UPDATE biddings SET status=1, selected_shop_id=选中店                     |
| 5   |              | INSERT merchant_messages (新订单已生成)                                      |


**结果**：竞价结束，订单生成并自动接单，服务商无需手动接单；选中店报价成交，其他店报价已失效。

#### 路径 B：用户主动结束（未选任何报价）


| 步骤  | 触发            | 数据变更                                                |
| --- | ------------- | --------------------------------------------------- |
| 1   | 用户点击「结束竞价」并确认 | UPDATE biddings SET status=1                        |
| 2   |               | UPDATE quotes SET quote_status=2 WHERE bidding_id=? |


**结果**：竞价结束，所有报价已失效，无订单。

#### 路径 C：竞价截止（expire_at 到达）


| 步骤  | 触发           | 数据变更                                                                              |
| --- | ------------ | --------------------------------------------------------------------------------- |
| 1   | expire_at 到达 | **不再自动关闭**；竞价 status 保持 0，已报价仍有效，用户可随时接受                                          |
| 2   | 服务商可见性       | **待报价** Tab：`expire_at > NOW()`，窗口外不可再提交新报价；**已报价** Tab：同一竞价在窗口结束后仍展示，直至车主选厂或结束竞价 |
| 3   | 报价失效         | 由 **报价有效期** quote_valid_until 控制；到期后该报价不可被用户接受                                    |


**说明**：竞价 2h 截止仅控制「能否继续收到新报价」；已提交的报价在 quote_valid_until 到期前均可被用户选择。

**报价有效期写入**：服务端使用 `DATE_ADD(NOW(), INTERVAL n DAY)` 写入 `quote_valid_until`，与库内 `NOW()` 比较一致；禁止再用 `toISOString()` 拼 UTC 字符串写入 `DATETIME`，以免与数据库会话时区错位导致「未满期却显示已过期」。

**历史误关单（仅测试/确认未手动结束竞价）**：若旧版逻辑在 `expire_at` 后将 `biddings.status` 置为 1、`quotes.quote_status` 置为 2，车主会无法选厂。可个案执行：`UPDATE biddings SET status=0, updated_at=NOW() WHERE bidding_id=? AND selected_shop_id IS NULL AND NOT EXISTS (SELECT 1 FROM orders o WHERE o.bidding_id=? AND o.status!=4);` 并对该竞价 `UPDATE quotes SET quote_status=0 WHERE bidding_id=?`。**勿**在生产对大量数据盲跑，以免恢复用户已主动结束的竞价。

#### 路径 D：订单取消后竞价重开

**撤单规则**（详见《订单撤单与维修完成流程.md》）：

- 服务商未接单：车主可随时撤销
- 服务商已接单 ≤ 30 分钟：车主可随时撤销
- 服务商已接单 > 30 分钟：需服务商同意或人工处理，且需填写撤单理由
- 订单已完成：不可撤销


| 步骤  | 触发                   | 数据变更                                                |
| --- | -------------------- | --------------------------------------------------- |
| 1   | 车主撤销订单（直接或经服务商/人工同意） | UPDATE orders SET status=4                          |
| 2   |                      | UPDATE biddings SET status=0, selected_shop_id=NULL |


**结果**：竞价重新进行中，用户可再次选厂。

### 3.5 车主端可见状态与文案（与 `biddings.status`、`expire_at` 的关系）


| 用户可见状态  | 数据条件                                   | 说明                                                                                 |
| ------- | -------------------------------------- | ---------------------------------------------------------------------------------- |
| **询价中** | `status=0` 且 `expire_at > NOW()`       | 竞价窗口内，仍可能有新报价；列表角标、与「未完成」Tab 一致。                                                   |
| **待选厂** | `status=0` 且 `expire_at <= NOW()` 且未选厂 | 窗口已截止，**库内仍为进行中**；已提交的报价在 `quote_valid_until` 内仍可选厂。列表**不得**再使用「已结束」指代本状态（易与关单混淆）。 |
| **已关闭** | `status=1`（或 `2` 已取消）                  | 用户已选厂成交、主动结束本轮、或重新发起询价时服务端关闭的旧轮次；不可再选厂。                                            |


**列表 Tab**：「未完成」对应 `status=0`（含询价中与待选厂）；「已关闭」对应 `status=1`（及按需包含已取消）。

**操作**：

- **结束本轮比价**：`status=0` 且未选厂时均可使用（含窗口内与窗口已截止），效果同路径 B。
- **重新发起询价**：同一份 `report_id` 再开一轮；确认文案需说明可能先结束当前轮并作废报价；服务端对「同 report、窗口已过、未选厂」会先关单再 `INSERT` 新竞价（见 `bidding-service.createBidding`）。

**竞价报价页**：倒计时展示「窗口已截止」表示仅询价窗口结束，不等于已关单；底部可提供与列表一致的「结束本轮比价」「重新发起询价」（满足条件时）。

**详情接口**：`GET /api/v1/bidding/:id` 返回 `report_id`、`selected_shop_id`，供前端展示上述操作。

## 4. 工作台与列表一致性

### 4.1 原则

**工作台「待报价」数量** 与 **竞价列表「待报价」Tab 的 total** 必须来自**同一套查询逻辑**。

### 4.2 统一筛选条件（待报价）

```sql
biddings.status = 0
AND biddings.expire_at > NOW()
AND users.latitude IS NOT NULL AND users.longitude IS NOT NULL
AND shops.latitude IS NOT NULL AND shops.longitude IS NOT NULL
AND haversine(用户, 店铺) <= biddings.range_km
AND NOT EXISTS (SELECT 1 FROM quotes WHERE bidding_id=b.bidding_id AND shop_id=?)
```

### 4.3 「已报价」Tab 筛选（窗口结束后仍可见）

与 4.2 相同，但 `**expire_at` 条件放宽为恒真**（`1=1`），仅要求 `b.status = 0` 且本店已存在报价记录，便于商户跟进「已出价、车主尚未选」的竞价。

实现：`merchantVisibleWhereFragment(..., { requireBiddingWindowOpen: false })`，仅用于 `listBiddingsForShop` 的 `quoted` 分支。

### 4.4 实现方式

将上述逻辑封装为 `bidding-service.js` 中的统一函数，dashboard 与 merchant/biddings 均调用，保证 count 与 list 一致。

## 5. 异常与边界


| 场景          | 处理                    |
| ----------- | --------------------- |
| 用户无位置发起竞价   | 创建成功，但服务商端不展示（距离无法计算） |
| 店铺无位置       | 不展示该店铺                |
| 竞价已结束仍尝试报价  | 400 该竞价已结束            |
| 本店已报价仍尝试报价  | 400 您已提交过报价           |
| 本店不在范围内访问详情 | 404 竞价不存在或未邀请您        |
| 用户结束竞价时已有报价 | 提示「现有报价都会作废」          |

## 6. 日志要点

关键节点需记录：

- 竞价创建：bidding_id, user_id, report_id, range_km, 是否有位置
- 报价提交：bidding_id, shop_id, amount
- 用户选厂：bidding_id, shop_id, order_id
- 用户结束：bidding_id
- 服务商列表查询：shop_id, status, 返回 count/list.length
