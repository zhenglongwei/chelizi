# 服务商订单详情页

## 基本信息

| 项目 | 说明 |
|------|------|
| 页面路径 | `/pages/merchant/order/detail/index` |
| 页面标识 | 订单详情、接单、更新状态 |
| 载体 | 微信小程序 |
| 入口 | 服务商订单列表点击卡片 |

## 页面功能概述

展示订单完整信息，支持接单（待接单→维修中）、**修改报价单**（多轮，非「二次终结」）、更新状态（维修中→待确认）、联系车主等。

**竞价流程**：用户选厂后订单自动接单（status=1），并冻结 **预报价快照**；**车主选厂即视为认可该预报价**。到店后服务商 **可选** 通过 **修改报价单** 提交改价（保险须绑定定损单），车主小程序确认后 **锁价**；未改价则一直以预报价为结算与展示口径。后续仍可再发起新一轮报价与确认（见《维修方案调整与确认流程》）。

**维修完成**：服务商提交完工材料后，后台会异步质检；其中**结算单/定损单为必传**。提交不阻塞，但若缺失结算单或结算金额与系统金额不一致，将转后台人工审核并要求补充/修正；审核通过后订单才进入待验收（status=2）。历史数据可仍含 `lead_technician`、`parts_verification` 字段。

## 页面布局结构（自上而下作业动线）

| 环节 | 模块 | 说明 |
|------|------|------|
| **概览** | 订单状态、工期计时器 | 待接单/维修中状态与剩余工期 |
| （提示） | 材料质检提示 | 服务商提交完工材料后，后台将核验结算单与金额一致性；缺失/不一致会转人工并要求补充，不影响服务商继续上传过程留痕与改价入口 |
| **车况与约定** | 车辆信息、**车主上传的车辆照片**（定损/报案环节 `images`）、报价与方案/建议、增值服务（与车主订单详情同构：类目标题 + 已勾选模板原文 + 免责）、订单规则（复杂度与佣金）、车主联系 | **不展示** AI 解析的损伤列表（避免与真实伤情混淆）；**只读上下文**；不在此区放置「修改方案/修改报价单」入口；报价区内用分区卡片样式区分各块 |
| **维修过程**（仅 `status=1`） | 三格进度条、入口与改价 | 顶部 **维修前 / 维修过程 / 完工** 三格进度（中间格同时认「维修过程」与「零配件验真」两类节点有照）。**更新关键节点** 进入子页 `/pages/merchant/order/repair-milestone`：ActionSheet 含 **维修前、维修过程、零配件验真、完工**；零配件验真单独节点，用主图位上传（1～8 张）及验真说明（选填），**不再**挂在维修过程节点下传 `parts_photo_urls`（新写入恒走 `parts_verify_process` + `photo_urls`）。主按钮在无「完工」留痕意图时展示 **「提交过程记录」**（仅 POST 里程碑、通知车主）；在草稿含「完工」图或服务端已有 `after_process` 留痕时展示 **「提交并完工」**：先提交流水，**仅当本次含完工节点或已存在完工留痕**时再弹窗确认 `1→2`；接口层亦要求已存在 `after_process` 里程碑后才允许待验收 |
| **说明**（`status` 2–3） | 分项质保 | 售后说明 |
| 底部固定 | 接单 / 填写过程与完工信息 | 维修中主按钮与「更新关键节点」同跳子页，在子页内按意图 **提交过程记录** 或 **提交并完工**（见上） |
| （补充） | 预报价/改价入口 | 金额与方案列表在「车况与约定」只读展示；**修改报价单** 仅在「维修过程」出现，条件同前（如 `final_quote_status≠1` 等） |
| （补充） | 修改方案入口 | 列表在「车况与约定」；**修改方案** 按钮仅在「维修过程」，老单兼容规则不变 |

## 功能点详细说明

### 功能点 1：接单

| 项目 | 说明 |
|------|------|
| 功能 ID | F-M-ORD-DETAIL-001 |
| 功能描述 | 服务商接单，订单状态 0→1 |
| 操作流程 | 待接单时点击「接单」→ 确认 |
| 预期结果 | 状态变为维修中；可通知车主 |
| 限制 | 仅 status=0 时显示；**竞价流程**下用户选厂即自动接单，订单直接 status=1，不显示此按钮 |

### 功能点 2：提交完工材料并进入待验收

| 项目 | 说明 |
|------|------|
| 功能 ID | F-M-ORD-DETAIL-002 |
| 功能描述 | 维修完成后，服务商标记为待用户确认 |
| 操作流程 | 维修中时进入「维修过程与完工」子页 → 按需上传过程与完工补充图 → 点击「维修完成」→ 确认 |
| 预期结果 | 提交完工材料后后台异步审核；审核通过后状态变为待确认（2）；凭证存入 orders.completion_evidence；车主端可「确认完成」 |
| 限制 | 仅 status=1；**仅 `final_quote_status=1`（有待确认的改价）时禁用**，文案「请等待车主确认报价」；`final_quote_status=0` 可直接完工（以预报价为准）；老单仍可按 repair_plan_status 处理 |
| 凭证要求 | 详见《订单撤单与维修完成流程.md》3.2 节 |

### 功能点 2a：修改报价单（多轮）

| 项目 | 说明 |
|------|------|
| 功能 ID | F-M-ORD-DETAIL-002a |
| 功能描述 | 到店后按轮次修改/提交报价单，推送车主确认；可多次迭代，非单次「终结」 |
| 操作流程 | 维修中时在页面「**维修过程**」区块点击「修改报价单」（`final_quote_status≠1` 等条件与历史一致）→ 跳转编辑页 `mode=final` → 提交 `POST .../final-quote`；保险单须带定损单附件 |
| 预期结果 | final_quote_status=1，通知车主；车主确认后为 2；业务允许后续再开新轮 |
| 详见 | 《维修方案调整与确认流程.md》 |
| UI 一致性 | 编辑页 `/pages/merchant/order/repair-plan-edit` 与 M05 首次「提交报价」共用样式表 `styles/merchant-quote-form.wxss`（`mqf-*`），维修项目含分项金额、项目质保等同一套字段与布局 |

### 功能点 2b：修改维修方案（历史兼容）

| 项目 | 说明 |
|------|------|
| 功能描述 | 未迁移老单保留原「修改方案」；新单禁用加价型 repair_plan 更新 |

### 功能点 4：联系车主

| 操作流程 | 点击联系车主 |
| 预期结果 | 调起拨号或客服 |

### 功能点 5：查看报价与车辆信息

| 项目 | 说明 |
|------|------|
| 功能 ID | F-M-ORD-DETAIL-005 |
| 功能描述 | 展示订单关联的报价、车辆、定损信息；**佣金比例、复杂度等级、订单分级** |

## 页面参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | string | 是 | 订单 ID（order_id） |

## 交互逻辑

1. **进入页面**：根据 id 加载订单详情，校验是否本店订单。
2. **状态驱动**：根据 status 展示不同操作（接单、进入「维修过程与完工」子页）。
3. **接单**：确认后调用接单接口。
4. **维修完成**：在子页内确认后调用更新状态接口（详情页仅入口）。

## 依赖接口

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/v1/merchant/orders/:id | GET | 获取订单详情（含 repair_plan、repair_plan_status） |
| /api/v1/merchant/orders/:id/accept | POST | 接单，status 0→1 |
| /api/v1/merchant/orders/:id/status | PUT | 更新状态，请求体 { status, completion_evidence }；维修完成(1→2)时 completion_evidence 必传；repair_plan_status=1 时禁用 |
| /api/v1/merchant/orders/:id/repair-plan | PUT | 修改维修方案 |

## 异常与边界

- 订单不存在或非本店：提示并返回。
- 重复接单：接口幂等或提示已接单。
- 车主已取消：展示已取消，隐藏接单等操作。
- 取消交易：车主端取消交易以“拆检留痕”为锚点；服务商端不再存在“撤单申请”响应入口（旧链路已下线）。
