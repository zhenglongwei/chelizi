# 主评价页

> **评价体系基准**：以 `docs/体系/02`、`03` 及页面索引为准；历史稿见 `docs/已归档/评价和激励体系.md`。本页整合全链路 4 阶评价中的**前 3 阶**（报价服务、维修过程、完工验收），服务完成后**1 屏到底、1 次提交**。

## 基本信息

| 项目 | 说明 |
|------|------|
| 页面路径 | `/pages/review/submit` |
| 页面标识 | 主评价、完工验收评价 |
| TabBar 位置 | 否 |

## 页面功能概述

**流程触发**：默认在「用户确认维修完工 + 平台完成分账结算」后弹出；后台配置 `require_settlement_before_review=0` 时，平台可提前弹出（提前返现）。不可跳过（可最小化，24 小时内再次打开 APP 强制弹出）。单页纵向滚动；**当前小程序主路径为极简 v3**（`review_form_version=3`）：**店头**；**核心指标标签行**（与五星序相关，**不含**外观百分比/配件 AI 短标签）；**请您评价**——**题面顺序**为 **① 报价透明度 1～5 星**（题干为「报价变动合理吗」等；折叠内为各**报价节点金额**`quoteDisplayNodes` + 结算/定损凭证图；多节点**金额一致**且用户打 1～2 星时弹窗引导在补充说明中填写**线下加价**等情况）**② 故障整体修复程度 1～5 星**（题干为「是否达到要求」**无**折叠照片，过程类照片在下一题）**③ 流程透明度 1～5 星**（题干为「是否及时了解维修过程」；折叠内**不展示**`quote_timeline` 价格线，仅**维修过程留痕**`repair_milestone_trace` 各节点**照片**`photo_urls`/`parts_photo_urls`）**④ 配件与材料可溯源度 1～5 星**（题干为是否知悉验真方法；折叠为**配件留档图**+**验真方案说明**+快捷验真，**不展示**配件 AI 归纳）**⑤ 服务态度 1～5 星 ⑥ 补充说明与图片（选填）**；`system_checks_preview` 与接口 `for-review` 一致，经 `sanitizeSystemChecksForUserFacing`：**报价节点**（`quote_flow.nodes` 同价合并规则同 `buildQuoteDisplayNodes`）、**结算单/定损材料图**，**不含** `quote_deviation`、`appearance`、配件侧归纳结论；`GET .../for-review` 的 `repair_milestone_trace.items` 含每节点 `photo_urls`、`parts_photo_urls` 供第③题；**客观题材料提示**（`objective_hints`）中 **`settlement_mismatch` 不在本页展示**；**快捷记录验真**（选填）；**分项公开授权**见 `review_public_media`；**本页车主上传**写入 `owner_always_public_urls`；可选 `ai_preview_fingerprint`（仅留痕指纹，不含外观/配件 AI 文案）。后台仍存完整 `review_system_checks` 并计算 `star_ai_anomaly`，与历史 **v2** 并存时以 02 文档 `version` 为准。

## 页面布局结构

| 模块 | 说明 |
|------|------|
| **顶部合规提示** | 固定展示，不可滚动隐藏：奖励仅与**完整性、真实性**挂钩，与好评/差评倾向无关；**符合规则时差评也可**按规则领取**基础**奖励。系统留痕材料仅供对照；各维星级需本人完成；平台**不对维修结果作鉴定结论**（与 C 端 `review-compliance-text` 一致） |
| **店头** | logo、店名、`shop_address_line`（区县+详细地址） |
| **核心指标标签** | 流程/报价/修复/配件/服务星（无系统/AI 归纳类短标签） |
| **请您评价（v3）** | **五星序** + 流程维折叠留痕 + 报价/维修/配件项可展开**订单留痕图与报价节点**（无外观 AI、无配件归纳文案）；`has_divergence` 可由低星或历史对齐项触发，管理端 `pool=ai_divergence` 可筛。`for-review` 返回 `valid_review_count`：当 **≥3** 时主区引导**缩短**（老用户少字说明） |
| **模块一 系统固化证据** | `GET /api/v1/user/orders/:id/for-review` 返回 `evidence_sections`（修前报价/过程/交付/质保四区 + 事故车定损区）、`review_scene`、`is_insurance_accident`、`objective_hints`（软提示，不替代审核）。质保区以**订单方案解析的分项质保约定列表**为主；历史单若有店端上传的纸质质保类补充图仍可展示，详见《02》与《分项质保 — 对外用语口径》（`docs/产品/电子质保凭证-对外用语.md`） |
| **报价协商过程公示** | **单卡合并**原「本单价格与方案」与「报价协商记录（公示）」，与模块一修前区「本单价格与方案摘要」、过程区「报价协商与变更记录」**去重**（有 `quote_proposal_history` 公示行时证据区不再重复点分摘要；有到店多轮时过程区不再重复轮次一行）。内容含：`quote_timeline` 说明/竞价快照/涨跌提示（若有）、**方案与金额对照**（`planCompare`，左侧优先 `quote_plan`）、**公示列表** `quote_proposal_history`（含合成预报价行；默认一行摘要 `quoteProposalSummary`，多条或含证明图可展开）。文案与字段口径见 `utils/quote-nomenclature.js` / 接口 `quote_nomenclature`；时间格式见 `utils/quote-proposal-display.js`。提交评价后**店铺详情评价列表、评价流**与公示列表一致 |
| **系统核验** | 6 项由系统自动核验，用户不答：告知费用、额外项目、服务商一致、结算金额、工期、质保。核验以**锁价后**订单数据为准（`final_quote_snapshot` / `quoted_amount` 与 `actual_amount` 一致） |
| **模块二 必答客观题** | 自费 5 题 / 事故 6 题，题干与 [《重构评价体系》第二节（已归档）](../已归档/重构评价体系.md) 一致；字段见《02-评价内容质量等级体系》2.1 |
| **模块三 体验星级** | 选填；未选按默认参与核算；客观题选「否」时对应维度最高 2 星（前后端双校验） |
| **补充材料** | 选填。页底**一个**车主自主上传图片入口（不区分类别、最多 12 张），完工/结算/定损/举证等均通过该入口；不重复展示服务商已传材料；服务商结算/过程/完工图等在**模块一 系统固化证据**与折叠区中展示。提交时全部写入 `reviews.completion_images`；为与公示/AI 兼容，`settlement_list_image` 取**首张**补充图（无则空），`fault_evidence_images` 可为空数组。**隐私提示**：尽量避免清晰车牌、VIN、证件、结算单姓名电话；远景或局部特写更安全（页内文案引导） |
| **对外展示图片（必选）** | 主评价提交前**二选一**：是否允许在店铺评价列表、口碑流等**公示场景**展示本评价相关的全部图片（含维修前后、补充实拍、公示区可能出现的店端留档等）。写入 `reviews.review_images_public`（1/0），**提交后不可改**；**不参与**内容质量等级与奖励金计算。`POST /api/v1/reviews` 传 `review_images_public`（或与 `module3` 同语义字段，服务端优先读顶层） |
| **模块四 主观补充** | 选填展示；**（v3）**领取基础轨**不要求**主观长文（与《02》《03》一致）；**不填不影响基础奖**（须已完成五星序）。若有文字则须过反作弊/安全审核。若**带图**，审核侧重**严重事实矛盾**类核对，**不评**文字长短或文采。双引导（避坑 / 亮点）；详细指引抽屉 |
| 奖励金预估 | 按订单分级展示可获奖励金；**不以本页补充文字长短定档**；基础轨与客观题/订单/账号相关，其他互动/转化类另按规则。主评价按当时等级发放；首评入库内容等级最高 2，追评可整体重评升级，差额补发。展示「以**实际到账**为准」。详见《03-全链路激励驱动体系》2.2 |
| 提交按钮 | 固定在底部，实时显示完成提示；客观题与有效描述未齐时置灰 |

## 服务商凭证与评价预填

服务商维修完成时上传的凭证（`orders.completion_evidence`）在车主评价时**自动预填**到对应模块，**不可删除**，需明确标注「服务商上传」；车主可**新增**照片，标注「车主上传」。

| 服务商凭证 | 评价对应 | 用途 |
|-----------|----------|------|
| 定损单/结算单 | 模块 1、3 | 比较实际项目与预估报价，计算报价准确性（保险才有定损单，否则用结算单） |
| 修复后照片 | 模块 3 完工实拍图 | 与维修项目匹配的完工实拍 |
| 物料照片 | 模块 2 维修过程 | 材料、配件相关 |

详见《订单撤单与维修完成流程.md》3.4 节。

## 交互逻辑

1. **进入页面**：用户点击「确认维修完工」后全屏弹出（是否需等分账完成见流程触发）；根据 `order_id` 加载订单、维修厂、奖励金预估；若订单有 `completion_evidence`，自动预填到对应模块并标注来源。
2. **必填进度**：实时校验各模块必填项，底部按钮显示完成进度；未完成置灰，完成后可点击。
3. **提交评价**：校验必填项 → 调用 `POST /api/v1/reviews` → 若本单带有可审的**车主补充图 URL**，后端以千问做**服务商留档图与车主文字/客观题**的**明确矛盾**与**底线合规**（不因「文字过短/不优质」拦截；见 `analyzeReviewEvidenceContradictionWithQwen`）。
4. **审核结果**：
   - **通过**：首评奖励按**规则完成度**与订单维度计算（**不以**千问 `contentQuality` 档位定现金）；追评默认不触发 `upgrade_diff` 自动补差（见 `reward_rules.platformIncentiveV1.settleUpgradeDiffEnabled`）。跳转订单详情或个人中心。
   - **不通过**：标注明确缺项，引导用户补充，**无提交次数限制，无惩罚**；用户可修改后重新提交。

## 提交成功响应（扩展）

`POST /api/v1/reviews` 成功时 `data` 中：

| 字段 | 说明 |
|------|------|
| is_invalid | 为 `true` 表示评价已入库为**无效评价**（用户选择「仍要提交」），无奖励金；`reviews.ai_analysis` 存结构化原因与改进建议 |
| invalid_reason | 主因文案（与弹窗/校验口径一致） |
| improvement_hints | 字符串数组，端上逐条展示「如何完善」 |
| content_quality / content_quality_level | 无效时为 `invalid` / `1` |

运营后台「评价审核」列表同步展示 `content_quality_level`、`content_quality`、`reviews.status`（前台是否展示）及由 `ai_analysis` 解析的评级说明；可选审核池 **系统/AI 归纳分歧**（`pool=ai_divergence`），依赖 `review_system_checks.user_ai_alignment.has_divergence`（含低星快照等）。

## for-review 响应扩展

| 字段 | 说明 |
|------|------|
| order_verification | 系统核验结果：{ informed, no_extra_project, shop_match, settlement_match, on_time, warranty_informed } |
| order_verification_all_ok | 是否全部核验通过 |
| valid_review_count | 用户累计有效评价数，≥3 时评价页详细指引默认收起（老用户适配） |
| pre_quote_plan / final_quote_plan | 服务端已解析；用于方案对比与金额行（对应库表 `pre_quote_snapshot` / `final_quote_snapshot`；老单可仅从 `quote` + `repair_plan`/`actual_amount` 推导） |
| quote_timeline | **报价历程摘要**：`summary` 中「含二次…」段由 `getProposalRoundsSummaryLabel(到店报价条数)` 生成；**条数仅计 `order_quote_proposals` 行**，不含列表首部合成预报价行（避免 1 轮到店却显示「二次、三次」）。另含预报价金额、确认报价、结算、`bidding_prequote_caption`、`highlight`、`bidding_snapshot`、`proposal_rounds`（与上述条数一致） |
| quote_nomenclature | `{ stage_codes, labels }`，与小程序 `utils/quote-nomenclature.js` 对齐，减少端上硬编码 |
| bidding_quote_plan | 与 `quote_plan` 相同，语义为竞价阶段选中行的 `quotes` 快照（产品侧即预报价行） |
| bidding_id | 有竞价流程时不为空；用于判断是否展示「预报价（据照片与描述·无实物）」对比话术 |
| quote_proposal_history[] | 到店记录在 `order_quote_proposals`；接口在可展示预报价计划时于列表**首部插入**合成行 `is_synthetic_pre_quote`（`proposal_id=display_pre_quote`）；**无到店记录但可展示预报价金额/方案时**亦可仅返回该合成行。其余项含 `display_round_label`、`quote_stage_code`（`listFormatted` / `revision_no`） |
| item_warranties | 聚合自 `final_quote_snapshot.items[].warranty_months`（及老数据兼容），供展示「逐项目质保」 |
| loss_assessment_image_urls | 定损/保险材料图 URL 数组（来自订单 `loss_assessment_documents`），与 `evidence_sections` 事故区一致；评价页「报价透明度」折叠区与 `merchant_settlement_list` 一并展示 |
| system_checks_preview.quote_deviation | 锁价与结算相对差异；**与报价节点展示一致**：若 `orders.actual_amount` 未落库，按与锁价相同计偏离 **0%**（不再误报「金额不完整」）。`scope_note` 说明拆检增项等可能合理情形（规则计算，非千问输出） |
| completion_evidence.exterior_repair_analysis | 服务商完工凭证**材料审核通过**（或免审核直连完工）时，由服务端对比**定损/维修前图**与 **`repair_photos`** 调用千问写入；`for-review` 的 `system_checks_preview.appearance` **直接映射此对象**，评价页不再提示「提交评价后再算」。历史单无该字段时 `appearance.status=pending`，提交评价后仍可用车主补充图异步兜底（见 `review-service-minimal-v3`） |
| completion_evidence.parts_traceability_ai | **配件/物料照片**与**维修方案分项**（`repair_plan.items`，缺省则回退竞价 `quotes.items`）千问视觉比对结果；存 **`match_level`**（`full_match` / `basic_match` / `mismatch`）、**`user_conclusion`**、**`mismatch_reasons`**、**`analysis_process`**（完整推理）、**`raw_model_excerpt`** 等。`for-review` 经 `parts_delivery.ai_display_line` **仅下发一句车主结论**（不匹配时附原因摘要）；全文不出现在小程序。三档语义：完全匹配＝各分项在照片中均有合理对应且无明确矛盾；基本匹配＝多数可对上、少数因清晰度/合影无法确认但无明确矛盾；不匹配＝明确矛盾或多项换件无可见依据。 |

## 异常与边界

- 订单不存在或已评价：提示并返回。
- 是否需等分账完成：由后台 `require_settlement_before_review` 配置决定，未分账完成时可能禁止评价或允许提前返现。
- 审核不通过：展示缺项说明，用户可补充后重新提交，无次数限制。
- 照片/材料不足：按模块要求提示必传项。
