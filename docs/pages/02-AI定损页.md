# AI 定损页

## 基本信息

| 项目 | 说明 |
|------|------|
| 页面路径 | `/pages/damage/upload` |
| 页面标识 | AI 定损、照片上传定损 |
| TabBar 位置 | 否（可从 TabBar「定损」入口进入） |

## 页面功能概述

用户上传事故车照片，由 AI 分析生成定损报告，并基于报告发起竞价，邀请维修厂报价。

## 页面布局结构

| 模块 | 说明 |
|------|------|
| 照片上传区 | 至少 1 张、最多 8 张（车身 45 度角、损伤特写、车架号等） |
| 情况描述 | 选填，用于泡水、异响等无法通过照片体现的问题 |
| 拍摄引导 | 拍摄示例图和文字说明 |
| 分析前说明 | 步骤一仅含拍摄引导、事故照片、情况描述；**车辆信息在 AI 分析完成并展示定损报告后再出现**，`analyze` 请求不传预填车辆字段 |
| AI 分析按钮 | 上传完成后点击开始 AI 分析 |
| 分析进度 | 显示进度条 |
| 定损报告 | 明显/可能损伤与维修建议（**不展示**预估费用区间；`analysis_result.total_estimate` 仍可存库供排序/公允等后台逻辑）；**多车时**在上传页选择车辆后，发起竞价会将 `analysis_focus_vehicle_id`（如 `车辆2`）写入 `biddings.vehicle_info` 与 `damage_reports.vehicle_info`；定损报告详情、车主竞价页、商户竞价/订单侧 AI 报告**仅展示该车**的 `human_display` 与带 `vehicleId` 的 `damages`，不再混排全车 |
| 报告内车辆信息 | 品牌、车型等以 AI 识别预填，用户核对修改；发起竞价前须填车牌；多车时随 Tab 切换。**车牌**：输入框展示/可编辑的为 AI 识别车牌（若有）；**不**将绑定车牌自动写入输入框，以免与 AI 结果混淆。用户**聚焦车牌输入框**时，在下方展示资料中**绑定车牌列表**（`/api/v1/user/vehicles`，非 AI），点击填入；无绑定则不出该块。若 AI 返回车牌与**任一**绑定车牌均不一致，展示**提示文案**，不拦截、不自动覆盖 |

## 功能点详细说明

### 功能点 1：照片上传

| 项目 | 说明 |
|------|------|
| 功能 ID | F-DAMAGE-001 |
| 功能描述 | 用户上传事故车照片 |
| 操作流程 | 1. 点击上传区域 2. 选择拍照或从相册选择 3. 图片压缩（最大 5MB） 4. 上传到阿里云 OSS 5. 显示上传进度和缩略图 |
| 预期结果 | 最多 8 张；显示拍摄示例；上传后显示缩略图；支持删除和重新上传 |
| 备注 | 至少 1 张照片；可选填情况描述（泡水、异响等）；单张不超过 5MB；自动压缩至 1080p |

### 功能点 2：AI 定损分析

| 项目 | 说明 |
|------|------|
| 功能 ID | F-DAMAGE-002 |
| 功能描述 | 调用 AI 分析事故照片，生成损失报告 |
| 操作流程 | 1. 至少上传 1 张照片（可选填情况描述）2. 点击「开始 AI 分析」 3. 调用 `/api/v1/damage/analyze`（`vehicle_info` 为空对象）4. 展示结果页后填写/核对车辆信息 5. 发起竞价 |
| 预期结果 | 进度 0–100%；报告含：损伤部位、预估维修项目、参考价格区间；若有情况描述，AI 结合描述与照片综合分析 |
| 备注 | 分析约 10–30 秒；失败时提示重新上传；报告仅供参考，以维修厂报价为准 |

### 功能点 3：发起竞价

| 项目 | 说明 |
|------|------|
| 功能 ID | F-DAMAGE-003 |
| 功能描述 | 基于 AI 报告发起竞价 |
| 操作流程 | 1. 确认（可编辑）报告；**多车时**选择己方车辆并确认车牌/品牌/车型 2. 选择竞价范围（5km/10km/全市）、是否走保险 3. 若走保险，选择事故类型及对应保险公司 4. 点击「发起竞价」 5. 系统向符合条件的维修厂推送邀请 6. 跳转竞价报价页 |
| 预期结果 | 发起后跳转报价页，实时显示已报价商家 |
| 备注 | 竞价有效期 24 小时；至少邀请 5 家；用户可随时结束竞价 |

## 页面参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| report_id | string | 否 | 若有已保存报告，可传入继续发起竞价 |
| from | string | 否 | 来源页（如 index、bidding） |

## 交互逻辑

1. **上传照片**：上传 → 校验数量与必填类型 → 压缩 → 上传 OSS → 显示缩略图。
2. **AI 分析**：校验最少张数 → 调用 analyze 接口 → 轮询/SSE 获取进度 → 展示报告。
3. **发起竞价**：校验报告、范围、保险信息 → 调用 create 接口 → 跳转 `/pages/bidding/detail?id={bidding_id}`。
4. **事故类型与保险**：根据是否走保险展示对应选项；选择后带入竞价请求。

## 依赖接口

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/v1/damage/analyze | POST | AI 定损分析 |
| /api/v1/damage/report | POST/GET | 获取定损报告 |
| /api/v1/bidding/create | POST | 发起竞价 |
| /api/v1/upload/image | POST | 上传图片至 OSS |

## 请求参数示例

**发起竞价 insurance_info 结构：**

```json
{
  "is_insurance": true,
  "accident_type": "single|self_fault|other_fault|equal_fault|other_main|self_main",
  "insurance_company": "string"
}
```

## 异常与边界

- 照片不足/格式异常：提示补充或重拍。
- AI 分析失败：提示重试或重新上传。
- 竞价范围无维修厂：提示扩大范围或换区域。
- 编辑报告：用户可修改 AI 报告内容再发起竞价，修改内容需随竞价一并存储。
